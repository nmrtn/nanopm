## 1. Problem summary
The documents nanopm generates are often long and verbose — slow to read — and there's no way to adapt the verbosity level. Since the value of the wiki depends on the founder coming back and re-reading artifacts, long docs raise the cost of exactly the re-read that retention depends on.

## 2. Value to the user
### Job to be done
The founder wants to control how detailed a generated document is — a terse version when he's scanning, a full version when he's drafting — so re-reading stays cheap. Today every artifact comes at one (long) verbosity and he absorbs the reading cost.

### Where we fall short
**No verbosity control on generated docs**
Output is verbose by default with no dial; long docs are slow to read.
- "Sur les documents qui sont générés, souvent je trouve que c'est un peu verbeux, c'est long à lire ; j'aimerais bien pouvoir adapter mon niveau de verbosité." — self-interview, 2026-06-29 (Granola dc61e5c2)

## 3. Value to the company
Small but real, and it compounds against retention: the wiki's whole premise is a persistent artifact the user returns to. If returning means re-reading walls of text, the compounding payoff is taxed at every visit.
