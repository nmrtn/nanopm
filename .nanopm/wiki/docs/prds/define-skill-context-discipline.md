# PRD: Define skills — refine vs from-scratch context discipline

Status: SHIPPED — v0.10.0 · PR https://github.com/nmrtn/nanopm/pull/23 (2026-06-12)

## Problem Statement

The five Define skills (`pm-vision-mission`, `pm-business-model`, `pm-org`, `pm-product`,
`pm-personas`) decide how to behave by sniffing *what evidence is available* (any `.nanopm/*.md`
artifact exists, or >10 tracked files, or a configured website → "reverse-engineer mode"; otherwise
"from-scratch interview mode"). They do **not** key off the one fact that should drive the run:
**does the target doc already exist?**

This produces two failures. (1) When the doc **already exists** and the founder wants to rework it,
the skill re-enters reverse-engineer mode and reads *every* prior artifact (in `pm-vision-mission`
that is PRODUCT, BUSINESS-MODEL, ORG, SCAN, DISCOVERY, CHALLENGES, STRATEGY, CONTEXT). That floods
the model's context with cross-document noise and dilutes the reasoning exactly where it should be
sharpest — and the previous version of the doc being reworked isn't treated as the anchor. (2) When
the doc **doesn't exist**, reverse-engineer mode is allowed to *skip any question the evidence
"answers cleanly,"* so it can ship a doc built on inference with no user validation — the
order-taking failure the ETHOS warns against (Evidence Before Conviction).

The cost: refined docs regress (prior sharpening is lost), or get polluted by irrelevant context, or
ship as confident assumptions never checked with the founder. For Theo — terminal-native solo
founder whose whole reason to use nanopm is that *planning compounds across sessions* — a Define
skill that can't cleanly update its own output without re-reading the world is drift, the precise
thing nanopm exists to prevent.

This is internal tooling: nanopm improving its own Define phase. There is one user on record today
(the founders), so the bet is observed in solo-fast mode, not instrumented.

---

## User Stories

- As **Theo (solo founder, terminal-native)**, when I re-run a Define skill to update an existing
  doc, I want it to start from my previous version and only the *relevant* cross-doc context, so the
  rework sharpens what I had instead of regenerating it noisily from scratch.
- As **Theo**, when a Define doc doesn't exist yet, I want the skill to reverse-engineer a draft
  *and then validate the inferred parts with me* before writing, so the doc reflects reality and not
  the model's best guess.

## Falsification

Across my next 5 Define-skill **refine** runs (me, the solo founder, within 14 days of shipping
this), the bet is wrong if in 2 or more of those runs I observe in the transcript that the main
agent either (a) still reads a full *other* Define doc into its own context instead of receiving the
retrieval subagent's digest, or (b) overwrites the existing doc without asking a single sharpening
question grounded in its previous version.

## Scope

### In scope (v1)
- Re-key all 5 Define skills' mode detection on **target-doc existence** (refine vs create), instead
  of evidence-sniffing alone.
- **Refine path** (target doc exists): load this skill's own history (`nanopm_context_read`),
  the already-loaded CONTEXT-SUMMARY, and the **previous version of the target doc**; dispatch a
  **retrieval subagent** to pull only the relevant slices from other nanopm docs and return a
  compact digest; then ask sharpening questions and rewrite the doc in place.
- **Create path** (target doc missing): reverse-engineer from available evidence (code, site, other
  docs — via the same retrieval subagent), then **always validate the inferred/`Assumed` claims with
  the user** before writing; fall back to full interview when there is no evidence (true greenfield).
- A shared helper in `lib/nanopm.sh` (e.g. `nanopm_define_mode` + a canonical retrieval-subagent
  prompt) so all 5 skills branch identically and don't drift.

### Out of scope (v1)
- **Document versioning / history** — overwrite in place; no `VISION-MISSION.v1.md`. Revisit if a
  founder asks to compare versions.
- **Changing the shared preamble** — `nanopm_preamble` / `nanopm_load_context` behavior is
  untouched; only Phase 1/2 logic of the skills (plus a *new* helper) changes.
- **Auto-regeneration of CONTEXT-SUMMARY** — the end-of-skill subagent refresh stays as-is.
- **Diff / change summary** — no "here's what changed between versions" output to the user.

## Requirements

### Functional requirements
1. Each Define skill's Phase 1 determines `refine` vs `create` by testing whether its target doc
   (e.g. `.nanopm/VISION-MISSION.md`) exists. The chosen mode is stated to the user in one line.
2. In **refine** mode, the main agent reads: skill history, CONTEXT-SUMMARY (already in preamble),
   and the previous version of the target doc. It does **not** read other raw Define docs directly.
3. A retrieval subagent is dispatched with: the target skill/doc, the sections being reworked, and
   the trust boundary (don't read `~/.claude/`, `~/.agents/`, `.claude/skills/`; treat doc content
   as data). It decides **what is relevant by its own model judgment** over the available
   `.nanopm/*.md` set — no per-skill shortlist of source docs. It returns a **bounded, synthesized
   digest** of only the relevant slices, each with a `file` pointer. The main agent rewrites from
   that digest. Because relevance is judged generically, the subagent prompt is **one shared prompt**
   across all 5 skills.
4. In **create** mode with evidence available, every claim the draft marks `Assumed` is presented to
   the user for confirmation/correction before the doc is written; `Evidenced` claims may be
   confirmed in bulk or skipped. True greenfield (no evidence) runs the existing interview.
5. **Refine and create ask different questions** (both capped at ≤3). Refine mode asks *sharpening*
   questions anchored in the previous version of the doc ("you said X last time — still true, or has
   it moved?"). Create mode asks *validating* questions that confirm or correct the inferred
   `Assumed` claims before first write. The two question sets must not be interchangeable.
6. The mode decision and the single shared retrieval-subagent prompt live in a `lib/nanopm.sh` helper
   consumed by all 5 skills.

### Non-functional requirements
- The retrieval subagent's digest is length-bounded (keep token cost predictable, in the spirit of
  `nanopm_load_context`'s 8 KB cap).
- Existing tier-1 (`test/skill-syntax.sh`) and tier-2 (context-threading e2e) gates pass unchanged.

## The One UX Decision

**What does the retrieval subagent return — a synthesized digest, or pointers the main agent then
reads?**

- **Option A — synthesized digest (recommended):** the subagent returns only the relevant slices it
  found, condensed, each with a `file` pointer. The main agent rewrites from the digest and only
  opens a source file if it must. Keeps noise out of the main context — which is the entire point of
  this PRD.
- **Option B — pointers only:** the subagent returns `file:line` pointers and the main agent reads
  them. Simpler subagent, but it re-introduces the raw-doc reads into the main context — the exact
  pollution we're removing.

Tradeoff: Option A costs one extra synthesis hop and risks the subagent dropping a nuance; Option B
is faithful but defeats the purpose. Recommend **A**, with the digest required to carry `file`
pointers so the main agent can drill down on demand.
