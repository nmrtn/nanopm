## 1. What You're Actually Building

nanopm is a set of Claude Code slash-command skills that orchestrate a structured PM planning workflow inside the CLI. It reads project context, asks targeted questions, and produces a chain of local markdown artifacts: AUDIT.md → OBJECTIVES.md → STRATEGY.md → ROADMAP.md → PRD. The shipped v0.1.0 delivers the full 9-skill pipeline with a shared runtime and test suite. It is opinionated about process and assumes users want structure, not just speed.

---

## 2. Who You're Actually Building It For

The stated audience is solo founders and indie hackers. But the product's real fit is a more deliberate builder — someone who has product complexity worth planning, can answer 11 audit questions with real answers, and is willing to invest 20 minutes in a planning cycle before shipping. The typical "ship it fast" indie hacker is not that person. The early users are close collaborators, not strangers — which means the real target audience hasn't been tested yet.

---

## 3. The Biggest Strategic Gap Right Now

The goal is "validate the pipeline end-to-end" but the only users are friends. Friends will complete the pipeline out of goodwill, not because it solved a real problem for them. The gap: zero external validation data. Distribution worry is real, but the deeper problem is that without strangers using the tool in their actual workflow, you can't know whether the pipeline produces artifacts that change behavior — or just feel productive.

---

## 4. The Question You're Avoiding

**Do solo founders actually want a structured PM pipeline, or are they using ad-hoc ChatGPT prompts precisely because they've rejected the ceremony?**

The strongest assumption here is that the friction users feel is *output inconsistency* — and that structure fixes it. But most indie hackers avoid audit→objectives→strategy→roadmap→PRD workflows deliberately. They find it bureaucratic and misaligned with gut-feel, fast iteration. nanopm may be solving a problem the target user doesn't have. The test: find three users who completed the full pipeline, check the commit history the week after. If the shipped code doesn't match the PRD, the tool provided the feeling of planning without the behavior change — and has no retention loop.
