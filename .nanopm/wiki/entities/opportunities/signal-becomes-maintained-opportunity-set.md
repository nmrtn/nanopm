## 1. Problem summary
A solo builder gathers signal — feedback, interviews, support tickets, their own observations — but it never reliably consolidates into a single maintained set of user problems. Opportunities live in the builder's head or scattered across docs, notes, and tickets, so the same problem gets re-discovered repeatedly and nothing accumulates. Without one durable list, the builder can't see the full problem space they're choosing from.

## 2. Value to the user
### Job to be done
When the builder finishes gathering signal, they want to turn it into one clear list of the user problems worth solving — so they're brainstorming against a complete, deduped picture rather than whatever they happen to remember. Today the alternative is re-reading scattered notes each time, or holding the problem set in memory, which silently loses problems and re-surfaces ones already considered.
### Where we fall short
**Signal scatters instead of consolidating**
The builder has no single place where gathered signal turns into a stable, named set of problems; each planning session starts from raw notes again.

**The same problem gets re-discovered**
Without dedup, one user problem reappears under different wording across sessions, wasting effort and obscuring which problems are genuinely distinct.

**Problems aren't separated from solutions**
Signal arrives tangled with feature ideas, so the builder reasons about solutions before the underlying user problem is even named — narrowing the option space prematurely.

## 3. Value to the company
Directly serves the overarching goal of a closed, autonomous product loop: Phase 3A (signal → list of opportunities) is the hinge between discovery and planning. A maintained, typed, deduped opportunity set is also a strong instance of the strategic retention bet on schema-validated state — durable structured data the builder returns to, rather than throwaway output.
