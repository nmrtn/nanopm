# PRD: Wiki-canonical writes (retire the flat-file double-write)
Generated by /pm-prd on 2026-06-24
Project: nanopm
Status: DRAFT

---

## Problem Statement

nanopm's memory wiki is **read-canonical but not write-canonical**, and that asymmetry is the unfinished half of the migration. The loaders already prefer the wiki (`nanopm_load_context` reads `wiki/overview/company.md` then falls back to the flat `CONTEXT-SUMMARY.md`; same for `nanopm_load_plan`). But on the **write** side, every Define and Plan skill still writes its flat `.nanopm/<X>.md` as the *primary* artifact, and a regen subagent *copies* a view into `wiki/docs/` or `wiki/overview/`. The same fact now lives in two writable places that can drift, and the redesign's load-bearing rule — "markdown + git is the only source of truth," one source of truth per fact (memory-wiki-redesign.md §6) — is not actually realized.

Measured in the repo on 2026-06-24: **21 skills write flat `.nanopm/*.md` files; only 9 touch the `wiki/` tree.** The flat doc is primary for *all* Define/Plan clean docs; the wiki copy is a synthesis the maintainer must keep in sync by hand. Today the cost is paid by the single maintainer remembering to reconcile two representations and by the viewer's path coupling (NANOPM-WIKI.md §12 "change one, change both") being doubled — every artifact that exists in two places is two places the viewer can mis-read.

This is the natural next migration step after Phase 2 (#110/#111 shipped the schema + ingest engine). It is **distinct from** the roadmap NOW items, which deepen the wiki (entity fan-out, lint, index bounding) but leave the dual-write seam in place.

---

## User Stories

- As the **wiki maintainer (Guillaume)**, I want each skill to write its artifact to exactly one canonical location, so I never hand-reconcile a flat doc against its wiki copy.
- As a **skill author (Nicolas)**, I want a test that fails when a skill writes a stray top-level `.nanopm/*.md` outside the allowed-flat set, so the single-source discipline is *enforced*, not *remembered*.
- As a **viewer user**, I want the GUI and the CLI to read the same canonical source, so they cannot disagree about what the company believes.

**Anti-goals (does NOT count as success for v1):**
- Migrating `raw/` (connector pulls, interviews, the typed event log) into the wiki — these are immutable sources by design (§1) and stay flat. The typed event log is machine-validated JSONL; it *cannot* be a markdown wiki page.
- Touching `NANOPM-WIKI.md` itself — it is the schema/contract skills read as the rulebook (§1), deliberately flat, not a wiki page.
- A standing flat-export write path that re-creates the double-write — exports are **on-demand only**, never a second source written every run.
- Building multi-writer git-merge machinery (NANOPM-WIKI.md §11 defers it) or a search index (§6 #5, disposable, deferred).
- The entity fan-out itself (roadmap NOW items 1–2) — that deepens the wiki; this PRD makes the wiki the canonical *write target*.

## Scope

**Decision (2026-06-24, Guillaume): Option A — move everything generated into the wiki.** Retire the flat Define/Plan doc layer entirely; the wiki is the single representation. See The One UX Decision for what this costs and how sidecars/shareability are handled.

- Route **Define** (vision-mission, business-model, org, product, personas) and **Plan** (objectives, strategy, roadmap) skill writes so the **wiki is the only substrate** — content lands in `wiki/` entity pages (via `nanopm-ingest-agent`) and/or `wiki/docs/<skill>.md` views. The flat `.nanopm/<X>.md` doc layer is **removed**, not kept as a projection.
- Fold the reasoning sidecars into the wiki's native provenance: the Evidenced/Assumed calls and sources that lived in `.nanopm/reasoning/<doc>.md` become per-claim citations (§5) and entity-page `## What we know` / `## Open / superseded` sections (§4.2). The sidecar's *content* survives; its separate-file mechanism does not.
- Provide a single-file **on-demand export** (`nanopm-export <section>`) that renders a wiki section to one shareable markdown file when asked — generated, never written automatically, never an edit surface.
- Scaffold the wiki by default: `nanopm_preamble` calls `nanopm_wiki_ensure` so `.nanopm/wiki/` exists in every project without a manual migrate step.
- Enforcement test `test/wiki-canonical.sh`: fails if a skill writes a top-level `.nanopm/*.md` outside the allow-list (`NANOPM-WIKI.md` only).
- Update `viewer/Models.swift` (`PhaseMapper` + the `ReasoningFiles` mapping, which now resolves to per-page provenance, not a sidecar file) in lockstep for every moved path (§12). **This is the larger viewer change Option A implies** — coordinate with Nicolas, who owns the viewer coupling.
- Make `nanopm-migrate-to-wiki` idempotent and lossless so existing half-migrated projects (this repo, Forto, runningcoach) converge cleanly — including a one-time fold of any existing `reasoning/` sidecars into provenance.
- Remove the now-stale residue surfaced in analysis: `PLAN-SUMMARY.md` (and any remaining flat Define doc) once its content is confirmed in the wiki.

### Out of scope (v1)
- `raw/` migration — revisit never; these are immutable sources by design.
- Entity fan-out to people/features — owned by roadmap NOW items 1–2; revisit there.
- Multi-writer merge machinery — revisit when parallel ingest waves actually collide (§11).
- A BM25/vector search index — revisit only if `index.md` stops scaling (§6 #5).
- Build/Daily-Ops skills (breakdown, retro, standup, weekly-update, brainstorm) — revisit in v2 once Define/Plan proves the pattern.

## Requirements

### Functional requirements
1. Each Define/Plan skill writes its canonical content **only** into the wiki (`wiki/docs/<skill>.md` view and/or entity pages via `nanopm-ingest-agent`). No skill writes a hand-authored flat `.nanopm/<X>.md`.
2. Reasoning/provenance (Evidenced/Assumed, sources) is captured inline as per-claim citations (§5) and entity-page sections (§4.2) — not as a separate `reasoning/` sidecar file.
3. `nanopm-export <section>` renders a wiki section to one shareable markdown file on demand; it is generated output, never read back as a source.
4. `nanopm_preamble` guarantees `.nanopm/wiki/` exists (via `nanopm_wiki_ensure`) on every run.
5. `test/wiki-canonical.sh` enforces the allow-list (`NANOPM-WIKI.md` only at top level) and is wired into the tier-1 test set referenced in CLAUDE.md.
6. `nanopm-migrate-to-wiki` is idempotent and lossless: re-running converges, never duplicates, never loses a claim (supersede, don't delete — §5); it folds existing `reasoning/` sidecars into provenance as part of the one-time move.

### Non-functional requirements
- No regression in the prefer-wiki-else-flat fallback during the transition (a half-migrated project must still load).
- Cross-host: where no Agent tool exists (some Vibe/Codex contexts), the write path degrades to an inline update or a `stale: true` marker (§10) — never blocks the skill.

## The One UX Decision — RESOLVED

**What happens to the human-shareable single-file Define docs?** **Decided 2026-06-24 (Guillaume): Option A — everything moves into the wiki; the flat doc layer is retired.**

This overrides the original Option B recommendation. The trade Guillaume accepts: the wiki becomes the single representation (maximum compounding — a persona enriched by interview + intel + data on one page), at the cost that "send me your product brief" is no longer one file sitting on disk by default. Two consequences are handled explicitly rather than lost:

- **Reasoning sidecars** → folded into the wiki's native provenance. The Evidenced/Assumed calls and source citations become per-claim provenance (§5) and entity-page `## What we know` / `## Open / superseded` sections (§4.2). This is *richer* than the old per-doc sidecar — provenance is now per-claim. The viewer's "Reasoning" affordance points at the page's provenance instead of a sidecar file.
- **Shareability** → `nanopm-export <section>` regenerates a one-file markdown on demand when someone needs to be handed a brief. It is generated, never an edit surface, never written automatically — so it cannot re-introduce the double-write.

**Residual risk to watch:** Option A is the bigger viewer change (every Define/Plan path moves, plus the sidecar→provenance remap). If the viewer remap slips, the GUI silently mis-renders (§12). This is the single most likely way the feature ships broken — gate handoff on the viewer change landing in the same wave as each path move.
