## Mission

nanopm gives solo founders and PMs building with AI coding agents an AI PM OS that automates discovery, signal surfacing, and continuous strategy revision — so deciding *what to build next* stops being the human bottleneck.

---

## Vision (3-5 years)

The product team is 90% automated. The AI PM OS runs continuously alongside the AI coding agent: it scans connected sources (user feedback, app reviews, support, analytics, competitor pages), surfaces and validates signals, revisits strategy as evidence arrives, and shortens the build–measure–learn loop from quarters to days. The human PM only makes the decisions a human must make — context calls, ethical trade-offs, the no's that require taste — and the rest of the PM job (the scanning, synthesis, falsification, retro) is on autopilot. The way Claude Code automates writing software, nanopm automates deciding *what* software to write. The destination is the automated loop running *alongside* the coding agent; the interface/viewer surface is a means to that loop — instrumentation that makes runs, artifacts, and signal visible and steerable — not a competing destination.

---

## Core Values

- **Problem first** — we don't write a PRD until the user + the observable problem is named concretely. If the problem hides behind buzzwords, we push back, not draft.
- **Falsify before you commit** — every strategic bet must name what would prove it wrong, with a segment, a metric, and a timeframe. A skill refuses to complete on a claim it can't falsify.
- **Subtract before you add** — every roadmap and PRD must name what we're explicitly NOT building. "Interesting, maybe some day" is the default answer to new ideas; the no's are as strategic as the yes's.
- **Ship to learn** — in solo-fast mode, the build IS the experiment. We ship the cheapest real test, not a mockup, and trust qualitative signal from 3–5 real users over instrumented dashboards we don't have.
- **Adversarial honesty** — we name the question the team is avoiding and ask it out loud, even (especially) when we're diagnosing ourselves. Sharper diagnosis beats comfortable consensus.

The mission above is *offensive*: automate the PM loop so what-to-build-next stops being the bottleneck — compound a decision loop alongside the coding agent, the way Claude Code compounds a build loop. The README is not wrong, but it understates the ambition by one order of magnitude. The viewer prototype, the connector library, the four-phase pipeline, and the signal-aggregation thesis all already live at the offensive ambition level. The public positioning has not caught up — a question the next strategy run should address.

At the mission level, the audience stays deliberately broad — solo founders *and* PMs building with AI coding agents — and the product is one destination, not two: the automated PM loop alongside the coding agent, with the viewer as instrumentation toward it. Which form factor (CLI or app) is the primary distribution channel is a strategy decision, not a mission one.
