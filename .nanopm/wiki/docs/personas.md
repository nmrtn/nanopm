## Primary Persona — Solo-builder Sam

*A solo founder who is simultaneously PM + engineer + founder, shipping a product with an AI coding agent in their terminal.*

- **The moment:** They're in their terminal mid-sprint, staring at a half-finished feature, realizing they've been building on vibes — no clear sense of whether this is the right thing to build next, for whom, or what "done" actually looks like.
- **Job to be done:** Functional — run the full product loop (discover → decide → plan → build → learn) without leaving the environment where they code. Emotional — feel like a real PM process is running alongside them, not like they're winging it alone.
- **Today's workaround:** A Notion doc that goes stale after week two. A ChatGPT session with no memory of last month's decisions. A Linear backlog created reactively from whatever the last user DM said. Or nothing at all — just their own head and git log.
- **The switch:** They ship the wrong feature (or a competitor ships what they should have shipped), and the cost of ad-hoc product decisions becomes real. Trigger: "I already have Claude Code in my terminal — what if it could do the PM work too?" Anxiety: "Will this add process overhead, or actually save me time?"
- **Recognize them by:** They commit ≥3 skill outputs to their repo. They run `/pm-run` a second time unprompted. They invoke a skill that reads from a prior skill's output within 14 days — the habit loop has started.

## Secondary Persona — Small-team lead Mia

*A technical co-founder or lead developer at a 2–5 person startup with no dedicated PM seat — everyone builds, nobody owns "what to build next" formally.*

*(Not yet validated by any real users — see Provenance. Solo builders are today; small teams are next.)*

- **The moment:** Standup devolves into a prioritization debate. Someone finishes a sprint on something nobody else thinks mattered. The team needs a PM process but won't hire a PM for 18 months.
- **Job to be done:** Coordinate product decisions across a small team without adding a PM layer or forcing everyone into a heavyweight tool. Keep multiple builders oriented toward the same bet.
- **Today's workaround:** A shared Notion backlog that rots. Slack threads that never close. One person making unilateral calls that others silently disagree with.
- **The switch:** The shared Linear backlog has been "groomed" twice and is still chaos. Someone says "we need a PM." Anxiety: "A CLI tool — whose terminal runs it? How do we share state?"
- **Recognize them by:** Multiple contributors invoke skills in the same repo. The output gets wired to their existing Linear or GitHub Issues workflow via `/pm-breakdown`.

## Anti-Persona — who we are NOT building for

**Established-PM Pat — a product manager at a company with a dedicated PM role, a product process already in place, and a team of engineers behind them.**

- **Why they're tempting:** Pat speaks fluent PM — they'd use every skill, give detailed feedback, and feel impressed by the pipeline. They look like a power user and would probably tell their network about it.
- **Why we say no:** nanopm *replaces* the PM function; for Pat, it *duplicates* it. Their feedback would pull nanopm toward Jira integrations, stakeholder management features, team-level permissions, and approval workflows — all of which violate the CLI/terminal ethos and the "no multi-user features" commitment. The product would stop being sharp for the person with no PM and start being weak for the person who already has one.
- **Revisit when:** Small-team validation (Mia) is complete AND there is explicit, repeated inbound from PM-seat users asking for a terminal-native PM layer — not the other way around.

**Action:** When a feature request optimizes for Pat — richer stakeholder reporting, multi-user state, enterprise integrations — the answer is no without a re-prioritization conversation.

## The one bet

**Solo builders who code in a terminal (using an AI coding agent) want an autonomous, compounding PM loop — not a Notion page, not a ChatGPT session, not their own head.** If this belief is wrong, the entire product collapses: the terminal-first design, the CLI UX, the context-compounding memory, the "no web UI" commitment — none of it matters if the user doesn't want PM work to happen *here*.

**Reality vs. aspiration:** Today, the only person who has run the full pipeline unprompted is Nicolas — the maintainer. The target user *is* the org. This is excellent dogfooding and a genuine n=1 risk simultaneously. The gap between "one person validates this in their own project" and "10 strangers adopt the habit loop" (Objective 1, KR1–KR3) is the gap that needs closing before the personas harden.
