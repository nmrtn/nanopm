## Strategic Position

Targeting solo founders and indie hackers who use Claude Code and want structured PM planning that persists across sessions. Not targeting teams with dedicated PMs, enterprise users, or SaaS PMs. The competitive frame is "better than ad-hoc ChatGPT PM prompts" — not "replace Linear" or "replace Notion."

---

## The Bet

If the artifact quality after a single `/pm-run` is visibly better than what a user gets from ad-hoc ChatGPT prompts, users will return — because the context-compounding advantage (`.nanopm/` artifacts persist across sessions) becomes obvious only through repeated use.

---

## How We Win

1. **Context persistence** — `.nanopm/` artifact chain compounds across sessions. ChatGPT loses context every time. This is the only durable advantage that can't be copied with a single prompt.
2. **Zero-overhead entry** — one command from inside the CLI they're already using. No new tool, no browser tab, no context switch to start.
3. **Opinionated structure** — the pipeline eliminates the blank-page problem without requiring PM expertise. The questions are asked; the user just answers.

---

## What We're Saying No To

- **Not building a SaaS or web UI** — embedding in the Claude Code CLI is the product decision, not a constraint. Revisit only if distribution through the CLI channel proves structurally impossible.
- **Not adding more skills** — 9 skills is enough surface area to validate. Every new skill before validation adds unvalidated complexity.
- **Not broad distribution before quality is validated** — driving traffic to a leaky bucket (artifacts that don't change what people ship) wastes the one asset we have: organic word-of-mouth in the Claude Code community.

---

## The Risk

Solo founders using Claude Code do their PM thinking somewhere else (Notion, ChatGPT, Google Docs) — the terminal is for code, not planning. If planning location is habitual rather than incidental, the embedding is wrong regardless of output quality, and no amount of artifact improvement fixes the distribution problem.

---

## Challenged by adversarial review

**Core assumption:** Solo founders using Claude Code will move their PM planning into the CLI alongside their code — that the location friction is surmountable once output quality is good enough.

**What would falsify it:** Fewer than 3 of 10 users who report satisfaction with the output quality return to use any nanopm skill within 14 days — indicating location, not quality, is the real barrier.

**Cheapest test:** DM 10 solo founders currently using Claude Code and ask: "When you're planning what to build next, where do you actually do that thinking — in your terminal, or somewhere else?" If 7+ say "somewhere else" as a habit (not an incidental choice), the embedding assumption is broken before any distribution work begins. Do this before anything else in Q2.

**Response:** The challenge is valid and sharpens the named risk without breaking the strategy. The bet is still worth making — but the cheapest test becomes the *first* action of Q2, not a backup plan. If 7+ founders say planning happens elsewhere, the strategy pivot is narrow: reframe from "plan in the terminal" to "generate artifacts in the terminal, take them wherever you work" — the pipeline still works, the framing shifts. The open question: is the context-compounding advantage compelling enough when the artifacts are consumed outside the CLI?
