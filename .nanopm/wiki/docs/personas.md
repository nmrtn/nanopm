## Primary Persona — Terminal-native Theo

*Solo founder / indie hacker shipping a product with an AI coding agent, no PM, lives in the terminal.*

- **The moment:** Theo just shipped his third feature this week with Claude Code / Cursor / Codex. The agent did exactly what he asked. He has a nagging feeling he's been asking the wrong questions — building polish on top of an unvalidated bet — and his last "strategy doc" is a 4,000-word ChatGPT thread he can't find again. He's looking for a way to enforce PM discipline *inside the editor he already lives in*, not in another SaaS tab.
- **Job to be done:** *Functional* — catch wrong-direction work before he spends weeks shipping it; make the planning compound across sessions instead of evaporating with the ChatGPT thread. *Emotional* — feel that his roadmap is honest (a falsifiable bet, not vibes wearing a PRD's clothing) and that the system will say "no" when he's about to do something dumb.
- **Today's workaround:** ChatGPT for strategy → Notion or Apple Notes for the PRD → Linear / GitHub Issues for tickets → vibes for "is this the right thing?". Nothing knows the code; nothing carries the prior decision into the next session; the agent will happily ship whatever he describes. Cost: months of compounding wrong-direction output, and the time tax of bouncing four tools every planning cycle.
- **The switch:** Theo sees nanopm in a tweet, on HN, or in a peer's setup. He runs `curl … | bash` and types `/pm-audit` within the same session. **The anxieties he must overcome:** (a) "Another CLI tool I install and abandon" and (b) "Is this just ChatGPT-in-the-terminal with extra steps?" The second pipeline run within 21 days is the moment the switch *completes*.
- **Recognize them by:** Install via `setup`; runs `/pm-run` or `/pm-audit` within 24h of install; opens a second nanopm session within 21 days; appears in `~/.nanopm/projects/{slug}/state.jsonl` with at least two distinct skill runs. The 27 stargazers + 1 forker + 0 known full-pipeline runs on GitHub is the *negative* signal of this persona today.
- **Confidence:** **Evidenced** — codebase center of gravity (setup auto-detects agent CLIs only, ETHOS's `solo-fast` default mode, README's "lives in your editor", STRATEGY.md's named 10-founder terminal-comfortable control cohort). **Assumed** on the retention claim: zero pipeline runs have ever been measured, so we know who the *implied* user is, not who actually comes back.

## Anti-Persona — who we are NOT building for

**Non-builders evaluating nanopm as PM software** — PMs, product managers, founder-curious operators who want a Notion/Linear/ProductBoard replacement and do *not* (and will not) run an AI coding agent.

- **Why they're tempting:** They are a *familiar shape*. The PM-software TAM is enormous and well-understood; "PM tool for indie startups" is an easier pitch than "PM layer for AI coding agents." They'd happily install a viewer that promises structured audits / strategies / PRDs and would never have to know what Cursor is. They show up in feedback DMs asking "can I use this without Claude Code?" and the reasonable-sounding answer is "yes, we can make that work."
- **Why we say no:** nanopm only delivers value *inside an agent runtime*. Every skill is a structured prompt the agent executes against the codebase, your nanopm state, and the connectors. Without the agent, you have markdown files and bash scripts — not a product. Serving this persona means rebuilding nanopm as PM SaaS: a hosted editor, our own LLM calls, our own auth, a Notion-shaped product backlog. That kills (a) the cost structure (we'd compete with Linear on price), (b) the agent-native value prop (the whole point is that the agent and the PM layer share one context), and (c) the architectural bet (typed cross-session state vs. the codebase only makes sense when an agent is reading both). Optimizing the macOS viewer for this segment is the failure mode that's already half-pulling us off course — every "make the viewer prettier so PMs can use it standalone" decision pushes one click closer to PM SaaS.
- **Revisit when:** An agent-runtime-free version of the planning loop is shown to produce output that compounds — i.e., when someone runs `/pm-audit` *without* an LLM in the loop and the AUDIT.md is still useful 30 days later. Under the current architecture, this is structurally never. Re-open this only on a deliberate strategy pivot, not on the next "but a PM at $company asked for it" inbound.

**Action:** When a feature request optimizes for the anti-persona (a standalone viewer that runs skills without an agent, SSO/RBAC for teams, a public hosted instance, a PM-tool import flow), the answer is **no** without a re-prioritization conversation that explicitly names the strategy reversal.

## The one bet

**The CLI's only blocker is form factor, not value** — i.e., if Theo (terminal-comfortable, no form-factor excuse) won't voluntarily run a full pipeline twice within 21 days, then Dani won't either, and the whole "PM layer for AI coding agents" thesis collapses regardless of how polished the viewer gets. This is AUDIT.md §4 restated as a persona claim, and STRATEGY.md's 10-founder control cohort exists precisely to falsify it.

**Reality-vs-aspiration gap (mandatory note):** Theo is who the *code* is built for; Dani is who the *recent commit energy* (viewer polish, designed rendering, app icon, run buttons) is built for. The 27 stargazers + 1 forker today are presumed-Theo, but **zero** of them have completed a pipeline — so even "Theo uses it today" is a hypothesis, not a fact. The viewer-side audience is pure aspiration: the cohort hasn't been recruited, the viewer isn't public, no Dani has ever opened the app. This gap is the most honest thing PERSONAS.md can say about today.
