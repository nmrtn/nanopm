# PRD: Multi-writer git machinery for the wiki (DEFERRED)
Generated by /pm-prd on 2026-06-23
Project: nanopm
Status: DEFERRED — LATER (split out of memory-wiki-redesign v1)

## Problem
The wiki v1 keeps concurrent ingest/bookkeeper subagents safe by partitioning work: each writer owns its target page, so concurrent writes touch disjoint files, commute, and git auto-merges. Once many ingests routinely target overlapping pages (e.g. several sources updating the same persona/competitor at once), that single-writer-per-file guarantee breaks down and we need real isolation plus deterministic merges. The key insight from the llm-wiki thread: a clean git merge is not a correct merge — two agents can write the same fact in different words and git happily merges both, creating a semantic duplicate it cannot see. So the real need is dedup-safe, order-independent (commutative) writes on a stable claim identity, not just merge mechanics.

## Scope (when promoted to NOW)
### In scope
- Worktree-per-writer isolation so concurrent ingests never share a working tree (most agent runtimes already provide a worktree per run — lean on that before building anything).
- A deterministic merge driver for the rare same-region collision: sorted append for claim lists, last-writer-wins on structured frontmatter fields — mechanical and reproducible, NOT an LLM resolving the conflict.
- Grep-dedup on a stable claim identity (the source citation, per v1's provenance-keyed claims) — kept permanently as the semantic layer git cannot see, run before every write so commutative writes stay duplicate-free regardless of merge order.

### Out of scope
- Advisory locks / handover protocols — these are symptoms of contended multi-writer access; correct partitioning removes the need rather than coordinating around it.
- LLM-driven merge conflict resolution — reintroduces the nondeterminism the deterministic driver exists to eliminate, and re-opens the semantic-duplicate hole.

## Revisit trigger (what promotes this from LATER to NOW)
Promote when concurrent ingest contention becomes real and observed: ≥2 ingest agents routinely target overlapping pages in the same run, OR we observe ≥1 semantic duplicate or lost write per 10 ingests under parallel runs (visible in `git diff` on `.nanopm/wiki/`). Until then, single-writer-per-file partitioning covers it.

## Success criteria
- 0 lost writes and 0 semantic duplicates across N parallel ingests targeting overlapping pages (verified by `git diff` + a grep-dedup pass on claim identity).
- Merge outcome is order-independent: the same set of concurrent writes produces the same final page regardless of which writer commits first.
