# Product Strategy
Generated by /pm-strategy on 2026-06-10
Project: nanopm
Period: Q3 2026 (the form-factor proof quarter)

---

## Strategic Position

nanopm is repositioned from **"a CLI skill pack for power users"** to **"an agent-driven PM layer that (1) aggregates product signal from everywhere and (2) makes the runs, artifacts, and connected sources visible and steerable through an interface above the agent."**

The target widens from terminal-native power users to **solo founders and product people who are not terminal-comfortable** — the people who feel the PM-discipline gap but for whom "open Claude Code in a repo and navigate to a local .nanopm/ folder" is a wall.

This quarter is a **proof quarter for one specific belief**: that the wall is the *form factor*, not the *value*. Today there is no demand proven — 27 stars, 1 fork, 0 external issues, and zero retention ever measured on anyone (state is local, telemetry was removed). The quarter's job is to find out which wall is real — and because prototyping is now cheap, the proof itself takes the form of a throwaway build, not a mockup.

---

## The Bet

**The primary blocker to nanopm's adoption is the form factor — working only inside a terminal, inside a repo, with artifacts buried in a local folder — not the value of the planning it produces. A visibility/orchestration layer above the agent (manage connected sources, see runs and their outputs, browse generated artifacts, see a timeline, get guided on when and why to run each skill) will unlock adoption beyond power users.**

This is a deliberate, founder-chosen pivot. nanopm operates in `solo-fast` mode (its own v0.8.0 principle: when cost-to-build ≈ cost-to-fake, the build *is* the experiment) — so the bet is tested by shipping the smallest real prototype, not by faking it. The strategy below exists to keep that prototype honest: small enough to stay a test, controlled enough to tell form factor from value.

---

## How We Win

1. **Automated multi-source signal aggregation.** Pull user interviews, App Store reviews, Reddit, support tickets, and competitor pages from the 15 existing connectors and synthesize them into the planning — opportunity/gap/strength/weakness mapping that a PM would spend days doing by hand. — **Why this holds:** this is agent-shaped work that compounds; the connector surface (15 today) is already built, so it's a repositioning of existing capability, not a from-scratch chase. The moat is the *synthesis across sources*, which a single-source tool can't replicate.

2. **A visibility/orchestration layer above the CLI.** A macOS app that runs Claude Code behind it: see connected sources, the runs that happened and their outputs, the artifacts, a timeline; be guided on what to run when and why. — **Why this holds:** it removes the single biggest friction (artifacts buried in a folder, no sense of what to do next) for the non-terminal-native majority. If validated, it's the difference between a power-user toy and a product a PM can adopt.

3. **Adversarial gates + schema-validated typed state remain the rigor underneath.** The app surfaces the work; the gates guarantee the work is falsifiable and typed. — **Why this holds:** a pretty dashboard over ad-hoc prompts is copyable; a dashboard over a falsifiability-enforcing, typed PM state layer is not.

## What We're Saying No To

- **No rebuilding a tracker** — Linear / Notion / GitHub Issues already exist downstream. nanopm hands off; it does not become the execution board. Revisit only if users explicitly refuse to leave the nanopm surface to track work.
- **No hosting the user's code** — stay local-first. The app runs the agent locally and reads local artifacts; it does not become a cloud IDE. Revisit never (permanent guardrail).
- **No building the full app before the prototype reads positive** — the prototype is a throwaway test instrument, scoped to one question (does graphical access to runs/artifacts change retention?), not the first increment of the product. Revisit when both proof tests pass.
- **No Symphony-only positioning** — carried over from v0.7.0. Symphony is one of six handoff targets. Permanent guardrail.

## The Risk

Three failure modes, in priority order:

1. **Empty house.** If the value doesn't retain even the power users who already tolerate the terminal, no app — prototype or polished — will save it. The team has zero retention data on anyone, so this is currently unknown.
2. **Confounded prototype.** A novel, nicer-looking macOS app can retain users by *novelty and polish*, not by *form-factor removal*. A positive prototype read, on its own, does not prove the form-factor thesis — it could be Hawthorne effect on friends. The terminal-comfortable control is the disambiguator.
3. **Scope creep.** "Prototyping is cheap" silently becomes "let's build the whole app" before the signal is in. The prototype must stay minimal and explicitly throwaway.

**Action:** Run BOTH proof tests below before committing to the full app build. The prototype tests whether non-terminal access changes behavior; the terminal-comfortable control tests whether the value retains the segment with no form-factor excuse. If the control fails, the bet is wrong and the response is to fix the *value*, not to build more *app*. If the prototype passes but the control fails, trust the control.

## How We Measure Demand (and which wall is real)

The bet hinges on distinguishing "form factor blocks" from "value blocks." Two tests do that:

| Test | What it isolates | Method | PASS / FAIL | Timeframe |
|---|---|---|---|---|
| **Throwaway macOS prototype** (the build is the experiment — solo-fast) | Does non-terminal access change adoption/retention? | Smallest possible macOS app that runs Claude Code behind it: launches `/pm-run`, lists generated artifacts readably, shows a timeline of runs. No source-management UI, no multi-project, no polish beyond legibility. Put it in front of 5–10 non-terminal-native product people. | **PASS:** ≥40% complete a pipeline and voluntarily return for a 2nd run within 14 days. **FAIL:** they open it once and don't return. | 2–3 weeks |
| **Terminal-comfortable control** (falsification) | Whether value retains the segment with NO form-factor excuse | Re-enable retention instrumentation (memory-read). Recruit 10 terminal-comfortable solo founders / PMs, have each run a full pipeline on the CLI. Count voluntary 2nd full runs, no prompting. | **FAIL (bet wrong):** fewer than 3 of 10 (30%) initiate a 2nd full pipeline run within 21 days → value is the blocker, not form factor. | 21 days |

**Cross-read (this is the actual measurement):**
- Prototype PASS **and** control PASS → value holds *and* form factor was the blocker → build the app.
- Prototype PASS **and** control FAIL → ⚠️ the prototype is likely retaining on novelty/polish, not form-factor removal → do not trust it; fix value first.
- Prototype FAIL **and** control PASS → value is real but the prototype didn't unlock the non-terminal segment → the prototype was wrong, or the segment is smaller than believed.
- Both FAIL → the value is the blocker. No app saves this.

**Secondary / public proxies (later):** repeated engagement by the same non-friend user (2nd issue/PR, active fork) > stars velocity. **Baseline 2026-06-10: 27 stars · 1 fork · 0 external issues · 0 retention measured.**

**Telemetry stays OFF** until a qualitative signal justifies a rebuild — do not repeat the v0.4.0→v0.6.0 premature-infrastructure mistake.

## Challenged by adversarial review

**Core assumption:** The underlying planning value already retains the users who reach it — i.e., the only thing between nanopm and broader adoption is the terminal form factor, not whether agent-generated PM artifacts are worth returning to.

**What would falsify it:** Recruit 10 terminal-comfortable solo founders / PMs (the segment with no form-factor excuse) and have them run a full pipeline this month; if fewer than 3 of 10 (30%) voluntarily initiate a second full pipeline run within 21 days, with no prompting, the form-factor bet is wrong — the people for whom the terminal is a non-issue still didn't return, which means the value, not the UI, is the blocker.

**Cheapest test:** Originally a Wizard-of-Oz (hand-deliver artifacts outside the terminal, no build) to isolate surface-from-value. Superseded by founder call: in solo-fast mode, prototyping a minimal macOS wrapper is cheap enough that the build IS the experiment. The trade-off — a prototype introduces confounds (novelty, polish) the Wizard-of-Oz avoided — is mitigated by (a) keeping the prototype minimal (it adds *access* to existing artifacts, not new value) and (b) keeping the terminal-comfortable control as the arm's-length value signal. The cross-read matrix above is what restores the form-factor-vs-value separation.

**Response:** Two rounds of challenge did not overturn the founder's pivot — they armed it. The bet stays (form factor is the primary blocker). What changed: the proof is now a throwaway macOS prototype (not a mockup), gated by a control that catches the empty-house failure mode before the full app is built. **Open question still unresolved:** the prototype cohort and control cohort are both likely to include friends — warm bias is real on both sides, so a clean signal needs at least some part-strangers (recruit from r/ClaudeAI or the Symphony discussion). If the two tests disagree, trust the control — it's the one with no form-factor excuse.
