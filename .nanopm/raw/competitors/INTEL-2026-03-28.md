## Strategic implications

**gstack's /autoplan vs nanopm's /pm-run:** gstack now runs a CEO → design → eng review pipeline fully automatically. This overlaps with nanopm's `/pm-run` at the orchestration layer. However, gstack's pipeline is engineering-review-oriented (code quality, architecture). nanopm's pipeline is product-planning-oriented (market, objectives, strategy, roadmap, PRD). The overlap is real but the layers are different — gstack automates "is the code good?", nanopm automates "are we building the right thing?" Make this distinction explicit in nanopm's positioning.

**deanpeters' CC BY-NC-SA license is a moat blocker for commercial users:** Any team that wants to build a product using deanpeters skills hits a license wall. nanopm (MIT-equivalent via being in your own repo) has an opening here — same pedagogic quality, no licensing constraint.

**deanpeters is heading toward AI PM Orchestrator skills (Phase 6):** Their planned `agent-orchestration-advisor`, `ai-product-evals`, and `ai-product-orchestrator` workflow will directly compete with nanopm's audit/strategy/roadmap pipeline. This is 1-2 months out based on their Phase 6 status. You have a window.

**gstack's design-shotgun + visual mockup generation (0.13.0) is a category expansion:** They moved from "code review" into "design generation" in one release. If nanopm ever wants design-phase coverage, the pattern is proven. Not urgent — but worth watching whether this drives adoption away from pure-PM tools.
