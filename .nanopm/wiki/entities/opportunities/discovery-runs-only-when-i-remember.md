## 1. Problem summary
A solo builder doing PM in the same terminal as their code can only monitor competitors, user feedback, analytics, and communities when they consciously stop building and trigger a run. Discovery is ad-hoc and sporadic, so changes between manual runs — a competitor shipping, a feedback spike, a Reddit thread — go unseen until much later, if ever. The PM wants discovery to happen continuously, every day, without being prompted.

## 2. Value to the user
### Job to be done
When I'm heads-down building, I want the system to keep watching my market, users, and product signals for me, so that I don't miss something important just because I was focused on code. The alternative today is manually running each discovery source on whatever cadence I happen to remember — which in practice means rarely, and inconsistently across sources.
### Where we fall short
**No standing cadence — discovery is on-demand only**
Nothing runs unless the PM initiates it. There is no daily heartbeat that sweeps competition, feedback, analytics, and web/community sources, so coverage is gappy and depends entirely on the PM's memory and free time.

**Uneven coverage across sources**
Even when the PM does run discovery, they tend to check whichever source is top of mind, leaving others (e.g. communities like Reddit, or analytics) stale between sessions.

## 3. Value to the company
The user's overarching goal is a closed, autonomous product loop; "discovery runs every day on its own" is Phase 2 of that loop and a direct expression of the autonomy bet. Continuous, unprompted discovery is what differentiates an autonomous PM from a set of on-demand commands. Related: [[continuous-discovery-floods-me-with-noise]], [[loop-runs-itself-on-a-cycle]].
