# PRD: Close the Opportunity → Solutions → PRD loop in the viewer
Generated by /pm-prd on 2026-06-28
Project: nanopm
Status: DRAFT

---

## Problem Statement

The viewer renders the Opportunity Solution Tree's **Opportunity** node (ranked DB + detail pages)
and already renders the **Solution** node (`SolutionsUI.swift` — overview table + detail + back-link
to the parent opportunity). But the *edges* between them are missing in the GUI: a founder reading
an opportunity cannot launch `/pm-solutions` for it, cannot see "this opportunity now has 3
candidate solutions," and — most importantly — has **no way to choose one and flow into a PRD**
without dropping to a terminal. So the loop the v0.24.0 work set out to enable —
*opportunity → compared solutions → choose → spec* — breaks at exactly the two moments of intent
(start the brainstorm; pick the winner). The cost is a context switch and a re-typed slug every
time, which is the friction that makes the founder "pick the first idea in his head and jump
straight to `/pm-prd`" — the failure `/pm-solutions` was built to prevent (`pm-solutions/SKILL.md`).

The viewer already launches every other skill in place via the phase Overview Run buttons; the
opportunity detail is the one high-intent surface that doesn't, and the compare-and-choose payoff
has nowhere to land on the page where the decision is actually made.

---

## User Stories

- As **the founder reading an opportunity** (Guillaume / Nicolas), I want to launch `/pm-solutions`
  for *this* opportunity with one click (optionally with a one-line steer), so that I move from
  problem to compared solutions without leaving the GUI or re-typing the slug.
- As **that founder**, once the run finishes I want to see the candidate solutions **side by side on
  the opportunity page** — pitch, appetite, impact, riskiest assumption, cheapest test — so that I
  can compare them on the same axes and decide, instead of opening three files.
- As **that founder**, I want one obvious action to **pick a solution and spec it** (launch
  `/pm-prd` for it) so that the loop closes forward without a terminal.

| Criteria | How Measured | Target |

| The full loop runs in the GUI | From an opportunity, complete opportunity → solutions run → read candidates → launch a PRD, entirely in the viewer, no terminal | The path is doable end-to-end at least once during dogfooding |
| Each launched run is scoped to the open opportunity | Open 3 opportunities, launch from each, confirm the right slug resolves (no "ambiguous / list candidates" stop) and the running strip tracks per-opportunity (not all at once) | 3/3 correct; concurrent runs don't cross-light |
| Candidates are comparable without opening files | After a run, the opportunity detail shows ≥3 cards each carrying pitch + appetite + impact + riskiest assumption + cheapest test | All five axes visible per card |
| Choosing flows into a PRD | Click "Spec this →" on a candidate, confirm `/pm-prd <solution-slug>` launches and the PRD lands | PRD run starts scoped to that solution |
| What will be different in commits after this ships? | Review git log 7 days post-ship | New/changed Swift under `viewer/Sources/NanoPMViewer/`: `OpportunitiesUI.swift` (launch button + popover + a Solutions-candidates section + running/success banner in `OpportunityDetailView`), `SolutionsUI.swift` (`Solution` parser extended to riskiest-assumption + cheapest-test; a compare-card view), `RunManager.swift`/`SkillCatalog.swift` (per-opportunity run tracking key); and `.nanopm/wiki/entities/solutions/` + new `prds/` pages that trace back to opportunities opened in the viewer |

**Anti-goals:** A run that doesn't carry the open opportunity's slug. Two opportunities sharing one
run-tracking key (the current `INDEX.md` collision). Any write to `.nanopm/` from the viewer process
(the browser stays read-only — choosing is expressed by *launching the PRD*, never by mutating
solution frontmatter). A second run engine — everything reuses `RunManager`.

## Falsification

In the 21 days after this ships, **neither founder (Guillaume nor Nicolas)** launches `/pm-solutions`

viewer's Activity Monitor run history and in `git log` of `.nanopm/wiki/entities/solutions/` and
`.nanopm/wiki/docs/prds/`. If that holds, the in-GUI loop-close is not the friction that was blocking
solution-brainstorming, and the surface should be cut rather than polished.

## Scope

### In scope (v1)
1. **Launch button** on `OpportunityDetailView`'s header (`OpportunitiesUI.swift:321`) — e.g.
   **"Brainstorm solutions"** — opening a pre-launch popover (reusing `SkillRunButton.swift:58`)
   with an **optional** free-text comment and a one-click **Run** that works with the field empty.
   The command sent is `/pm-solutions <opportunity-slug>`; the comment rides the existing
   `userContext` "CONTEXT FROM THE USER" channel (`RunManager.swift:233`).
2. **Per-opportunity run tracking** — launch tracks against an opportunity-scoped key, not the shared
   `wiki/entities/solutions/INDEX.md` path the catalog uses today (`SkillCatalog.swift:211`,
   `RunManager.swift:210`), so concurrent solutions runs on different opportunities don't collide.
3. **Lifecycle on the opportunity page** — an inline running strip (sparkle + last activity +
   elapsed, mirroring `OpportunitiesUI.swift:216`), an **"Answer…"** affordance routing mid-run
   questions to the existing `QuestionForm` (`RunSessionView.swift:155`), and on completion a
   **success banner** ("3 solutions ready"). **No auto-navigation** — pull, not push; the existing
   completion notification fires.
4. **Solutions-candidates compare section** appended to `OpportunityDetailView` (after the body,
   `OpportunitiesUI.swift:371`): the solutions whose parent is this opportunity, rendered as ≥3
   side-by-side cards showing pitch, lens·appetite, impact, riskiest assumption, cheapest test, with
   `[Open]` and `[Spec this →]` per card.
5. **Extend the `Solution` parser** (`SolutionsUI.swift:10`) to pull riskiest-assumption and
   cheapest-test (same one-line-under-heading technique as `summary`) so the cards are comparable.
6. **"Spec this →"** launches `/pm-prd <solution-slug>` via `RunManager.launch`. **Launching the PRD
   is the act of choosing** — no frontmatter write. The PRD run shows the same lifecycle.
7. **"N solutions" count badge** on the opportunity (detail header and ideally the ranked-table row,
   `OpportunitiesUI.swift:264`) — the forward edge, so the founder sees which opportunities are
   already worked into solutions. (Cheap reverse-lookup; include if it fits the cycle, else defer.)

### Out of scope (v1)
- Writing `status: chosen` (or any frontmatter) back to the solution file — the viewer stays
  read-only; if a "chosen" chip is wanted, let `/pm-prd` stamp it when it picks the solution up.
- A dedicated full-window compare screen — the on-detail section is enough; no second destination.
- Shortlisting / drag-to-rank / annotations on candidates, and diff/merge between solutions.
- A launch button on the ranked-table rows (detail-only for v1).
- Re-running / resuming a prior solutions run from the button — the Activity Monitor owns in-flight
  runs.

## Requirements

### Functional requirements
1. Render a launch `ActionButton` (`ActionButton.swift:89`) in `OpportunityDetailView`'s header;
   on click, present the optional-comment popover; on Run, call
   `RunManager.launch(_:in:userContext:)` with the command resolved to `/pm-solutions <slug>` and
   the trimmed comment as `userContext`. Slug derives from the opportunity file stem
   (`OpportunityFiles`, `Models.swift:175`).
2. Track the launched run against an opportunity-scoped key so `isActive`/`latestRun`
   (`RunManager.swift:210`) answer "is a solutions run in flight for *this* opportunity?" correctly
   under concurrency.
3. Reflect run state on the opportunity page: running strip → "Answer…" on `waitingForInput`
   (route to `RunSessionView`) → success banner on `succeeded` → inline error on `failed`
   (browsing unaffected, same path as other Run buttons). No auto-navigation on completion.
4. Query solutions whose `opportunity` matches this opportunity (reverse of
   `parentOpportunityArtifactID`, `SolutionsUI.swift:98`) and render them as the compare-card
   section; degrade gracefully to "no solutions yet" before the first run.
5. Extend the `Solution` model/parser to expose riskiest-assumption and cheapest-test for the card.
6. "Spec this →" calls `RunManager.launch` for `/pm-prd` scoped to the chosen solution's slug; on
   completion, surface a link to the new PRD doc.
7. Show a "N solutions" count on the opportunity (header; table row if it fits).

### Non-functional requirements
- No new dependency, no second process path, no write from the viewer process to `.nanopm/`.

## The One UX Decision

**How "choosing" a solution is expressed — and whether the viewer ever writes to disk.**

- **Option A (chosen): launching the PRD *is* the choice.** "Spec this →" launches
  `/pm-prd <solution-slug>`; the existence of the PRD (and the solution's own `status: speccing`
  transition that `/pm-prd` writes from its side) is the record. The viewer never mutates a solution
  file, so its read-only charter holds. The action is named by its consequence ("Spec this →"), which
  also sidesteps the "what does Choose even do?" ambiguity.
- **Option B (rejected): the viewer writes `status: chosen` to the solution frontmatter** to show a
  chip. Visually tidy, but it gives the read-only browser a write path — a real charter departure for
  a throwaway prototype, and redundant once `/pm-prd` records the transition anyway.

The tradeoff: Option A means the "chosen" state isn't visible until the PRD exists / the skill stamps
it — accepted, because keeping the GUI read-only is worth more than an instant chip. Getting this
wrong (letting the viewer write frontmatter) trades the prototype's cleanest invariant for cosmetics.

## Reviewer notes

*Advisory — surfaced by the /pm-prd review panel + designer review. Read before handoff.*

- **success-measurability:** The viewer has no analytics; counts are eyeballed by two founders. Treat
  the falsification's binary signal (does the loop get used *at all* — button launched, PRD spec'd —
  in 21 days?) as the real go/no-go; the per-criterion targets are soft.
- **persona-fit:** This proves the *founders'* loop is faster end-to-end; it does **not** by itself
  prove the non-terminal-native target persona can run discovery without CLI fluency (the user still
  reasons about opportunities, slugs, and launching agent skills). Keep that as the open question the
  broader form-factor bet must answer — don't read button-usage as persona-validation.
- **designer (plumbing):** The `pm-solutions` catalog entry tracks against a single
  `solutions/INDEX.md` path, so two opportunities launching solutions collide on one run key. Fix the
  tracking key first (req. 2) — it's ~20 lines and everything visual on the opportunity page depends
  on it being honest.
- **designer (scope):** Solutions already render in the viewer (`SolutionsUI.swift`) — do **not**
  rebuild a compare screen; the new section reuses that surface's model + tints. The genuinely new
  build is the on-opportunity compare section, "Spec this →", the parser extension, and the tracking
  fix.
