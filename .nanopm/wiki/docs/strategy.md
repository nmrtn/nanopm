# Product Strategy
Generated by /pm-strategy on 2026-06-26

## Strategic Position

nanopm plays in **the PM layer for AI-assisted product development** — the work between "I have an idea" and "I'm writing tickets in Linear." Primary user: solo founder or small team running Claude Code, Vibe, or Codex with no dedicated PM. NOT for established product teams with a PM seat, not for builders whose primary planning surface is a GUI tool (Notion, Linear).

The architectural claim: PM artifacts (bet, falsification, scope-outs, roadmap targets, PRD) are valuable independent of which downstream you use to deliver the work — and they get more valuable as context compounds across sessions. This is the differentiation that a one-shot ChatGPT session can't replicate.

---

## The Bet

**Getting ≥3 external users to complete a full nanopm pipeline and return for a second session is the single most important experiment for Q3.** If the habit loop is observed in the wild — by someone who was not told to return, who came back because the context compound made the second session noticeably better than the first — then the product thesis is confirmed and every downstream automation investment (autonomous loop, continuous discovery) is justified. If no external user returns unprompted within 60 days of a genuine distribution attempt, the problem is either a demand problem (the persona doesn't exist at scale) or a product problem (the habit loop isn't legible enough to sustain), and both require a fundamentally different strategy.

---

## How We Win

1. **Plugin marketplace as zero-friction entry.** The Claude Code plugin marketplace gives access to the right user at the moment they're already in-terminal with their AI coding agent. No marketing needed — the install is one click for the user who already lives in this environment. — **Why this holds:** this distribution channel is a structural fit that generic PM tools (Notion, Productboard) cannot replicate; they are GUI-first and sell to managers, not builders.

2. **The viewer as a "finished product" signal.** The macOS viewer converts a CLI skill pack into something that looks deployed. Solo builders are skeptical of tools that look like someone's side project; the viewer closes that gap in the first 30 seconds. — **Why this holds:** building a native viewer is a commitment that competitors offering one-shot prompts haven't made; it signals staying power.

3. **Context-compounding as a retention moat.** Once 3 sessions compound on each other, switching cost rises organically — lost context. A user who returns to nanopm for a second strategy run gets the prior bet, the prior risk, the prior scope-outs surfaced automatically. A ChatGPT session started cold can't do that. — **Why this holds:** the compounding is a product property, not a feature flag; competitors can't replicate it without rebuilding the entire wiki + state layer.

---

## What We're Saying No To

- **Not building the autonomous PM loop yet** — until ≥2 external users return for session 2 unprompted, automating session 3+ is optimization without a baseline. Revisit when Objective 1 KR2 (3 users say it changed what they built) is green.
- **Not expanding handoff targets beyond the current 6** — six targets exist; the bottleneck is distribution, not handoff breadth. Revisit when the first non-maintainer user requests a specific target that isn't covered.
- **Not pursuing monetization or a revenue model** — picking a price before 10 returning external users is a distraction. Revisit at 100 confirmed installs by strangers.
- **Not waiting for passive marketplace discovery to prove itself beyond 14 days** — if zero organic installs by day 14 post-listing, shift immediately to active Claude Code Discord seeding; passive-only distribution is the risk, not a patient strategy. Revisit if the Discord seeding also produces zero sessions.

## The Risk

The Claude Code plugin marketplace may have low enough discoverability that zero organic installs arrive in 60 days — meaning the assumed distribution channel is wrong before the habit loop can even be tested. There is no identified fallback GTM channel beyond Discord seeding.

**Action:** Publish the plugin to the marketplace this week. At day 14, check install count (marketplace dashboard or GitHub clone events). Zero organic installs by day 14 triggers immediate pivot to active Discord + community seeding — do not wait for day 60 to declare the passive channel failed.

## Challenged by adversarial review

**Core assumption:** The plugin marketplace will generate enough organic discovery that solo builders install and complete a full pipeline without active outreach from the founder.

**What would falsify it:** Fewer than 3 users from the Claude Code plugin marketplace (excluding the founder and anyone directly DM'd or referred) complete all four pipeline phases — Define, Discover, Plan, Build — within 60 days of the plugin listing going live.

**Cheapest test:** Publish the plugin this week and do nothing else — no posts, no DMs, no Reddit threads. Check install telemetry (or GitHub stars/clones if telemetry isn't wired) at day 14. Zero organic installs by day 14 confirms the marketplace-as-distribution assumption is wrong before another sprint is spent on it. Evidence format: raw install count in the marketplace dashboard or unique caller sessions in git log.

**Response:** The challenge is valid and sharpened the strategy. "How We Win #1" now explicitly distinguishes passive discovery (the cheapest test, run first) from active community seeding (the fallback, triggered at day 14 if passive fails). The core "distribution before features" bet is unchanged — the automation roadmap (autonomous loop, continuous discovery) remains gated on observing the habit loop in the wild.
