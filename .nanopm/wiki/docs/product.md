## What This Product Is

nanopm is a pack of **24 markdown "skills"** plus a shared bash runtime and a set of Python CLIs that turn an AI coding agent (Claude Code / Mistral Vibe / OpenAI Codex) into a PM. Each skill is a structured prompt the agent executes: it reads what's known, asks targeted questions, reads the codebase or site, and writes a human markdown artifact into a **maintained memory wiki** at `.nanopm/wiki/`. The next skill reads from there — the pipeline compounds across sessions. Memory is an **LLM-wiki** (Karpathy's pattern: three layers + three ops — ingest / query / lint), not a chat log. As of v0.23.0–0.24.0, the wiki now hosts a working **Opportunity Solution Tree** (Teresa Torres): Outcomes → Opportunities → Solutions → PRD, with planning skills reading the ranked opportunities and `/pm-solutions` seeding `/pm-prd <chosen-solution-slug>`. A second surface, a SwiftUI macOS viewer at `viewer/`, reads the same artifacts and exposes them as a nav-and-detail GUI with in-app skill runs.

## The Core Problem

PMs and solo founders building with AI coding agents lose the planning loop the moment they leave the editor. Strategy lives in ChatGPT, the PRD in Notion, tickets in Linear — nothing compounds, nothing knows the code, and the agent will happily ship whatever you describe without ever asking if it's the right thing. The pain hits every planning cycle: the team builds the wrong thing fast, then ships polish on top of an unvalidated bet. The cost is months of compounding wasted output. Inferred from README ("Building the wrong thing fast is the failure mode AI coding agents make easy") and from ETHOS.md's six principles, which the skills enforce as gates.

## The Opportunity Solution Tree (spine since v0.24.0)

Since v0.23.0–0.24.0, the planning substrate has a named shape: a **Teresa-Torres Opportunity Solution Tree** rooted in the wiki. The chain is wired end-to-end:

1. **Outcomes** — OKRs in `.nanopm/wiki/docs/objectives.md`, written by `/pm-objectives`.
2. **Opportunities** — ranked user problems under `.nanopm/wiki/entities/opportunities/`, written by `/pm-opportunities` (one file per opportunity + INDEX + LOG + SCHEMA, two levels only — Theme → Opportunity, no scoring at v1).
3. **Solutions** — three-lens brainstorm (Eng / Design / Business, concurrent), converged into ≥3 framed solutions per opportunity (pitch · appetite · impact · riskiest assumption · cheapest test · originating lens · dissent note), under `.nanopm/wiki/entities/solutions/`. Written by `/pm-solutions <opportunity-slug>`.
4. **PRD** — `/pm-prd <chosen-solution-slug>` consumes the chosen solution as its input.

Planning skills (`pm-objectives`, `pm-strategy`, `pm-roadmap`, `pm-discovery`) now query the ranked opportunities (top open, addressed?) instead of just raw feedback themes. `nanopm_plan_brief_prompt` reads `wiki/entities/opportunities/INDEX.md`, so the always-loaded plan brief carries "Top open opportunities." `/pm-opportunities` regenerates the brief on exit. This loop closure is gated by `test/opportunity-loop.sh`; the Solutions chain is gated by `test/wiki-canonical.sh` and three new lint rules (orphan solution / missing assumption-test / `chosen`-without-PRD).

## Surface Area & Main Features

**Two product surfaces over one maintained memory wiki.**

### Skill pack — 24 skills (all `pm-*/`)

- **Define (5)** — `/pm-vision-mission`, `/pm-business-model`, `/pm-org`, `/pm-product`, `/pm-personas`. Each is dual-mode: reverse-engineer from code + site when material exists, interview when greenfield. `pm-product` absorbs the retired `/pm-scan`.
- **Discover (6)** — `/pm-discovery`, `/pm-user-feedback`, `/pm-interview`, `/pm-data`, `/pm-competitors-intel`, `/pm-opportunities`. Pull external signals: user research, analytics, market intel, the ranked opportunity DB.
- **Plan (4)** — `/pm-objectives` → `/pm-strategy` → `/pm-roadmap` → `/pm-prd`. Adversarial gates fire on strategy / roadmap / PRD (and `/pm-solutions`, `/pm-challenge-me`, `/pm-breakdown`) — refuse completion until a falsifiable bet (segment + behavior + metric + timeframe) lands.
- **OST node — Solutions (1, new in v0.24.0)** — `/pm-solutions <opportunity-slug>` is the Solutions node of the OST chain above. Three concurrent expert lenses (Eng / Design / Business) → converged ≥3 framed solutions, written under `wiki/entities/solutions/`. New `solution` type in ingest / migrate / state-log; gated by `nanopm_state_log`.
- **Build / handoff (2)** — `/pm-breakdown` (six peer targets: Linear, GitHub, OpenSpec, gstack, Symphony, Human markdown), `/pm-retro` (roadmap NOW vs. actual commits).
- **Daily ops (4)** — `/pm-standup`, `/pm-weekly-update`, `/pm-brainstorm`, `/pm-challenge-me`. Parallel to the pipeline.
- **Orchestration / meta (2)** — `/pm-run` (full pipeline), `/pm-upgrade` (auto-detects install type).
- **Recipe form (engine + recipes).** Every skill reads upstream context through a shared **query** (or Define-scoped **retrieval**) subagent, reasons, and writes back — query → reasoning → ingest. No skill keeps bespoke per-skill read logic.

## The Core Workflow

1. Install via `setup` (auto-detects Claude Code / Vibe / Codex; installs skills + the shared runtime/CLIs to the host path and `~/.nanopm/`). On the first post-upgrade run, a project with legacy flat docs auto-migrates into the wiki; briefs self-heal if empty.
2. Run `/pm-run` (or any single skill standalone) inside the agent. The preamble loads the two always-on briefs (company + current-work, which now includes top open opportunities) and the wiki catalog.
3. Each skill **queries** the wiki for the context it needs, reasons, and **writes its artifact directly** into `.nanopm/wiki/` (gate-free); PM decisions also append a typed JSONL record.
4. Adversarial gates fire on strategy / roadmap / PRD / solutions — the skill won't complete unless a well-formed falsifiable record lands.
5. The OST chain compounds: `/pm-opportunities` ranks problems → `/pm-solutions <slug>` produces ≥3 framed solutions for one opportunity → `/pm-prd <chosen-solution-slug>` writes the spec → `/pm-breakdown` hands off to one of six peer targets.
6. The judgment lint runs daily (when due) to surface contradictions/gaps into `wiki/log.md`.

## Technical Bets

- **Markdown-as-program (SKILL.md standard).** The whole product is structured prompts the agent executes. Bet: a well-structured prompt plus hard validators is enough to enforce PM discipline across hosts. Risk: if LLM adherence drifts, the gates are the only backstop, and gate quality depends on subagent judgment.
- **The maintained LLM-wiki + the OST chain IS the planning substrate bet** *(extended 2026-06-29)*. The durable value is a curated, compounding memory wiki (Karpathy's ingest/query/lint, written freely and kept honest by an after-the-fact judgment lint) **plus** the Opportunity Solution Tree (Outcomes → Opportunities → Solutions → PRD) that gives planning a named shape. The typed JSONL state layer is the **decisions record under** the wiki. Memory was step 1 of this bet; OST is step 2. Risk: the safety net is a judgment lint whose real catch-rate is unproven (3-week falsification); the OST end-to-end loop is wired but no founder has run a solution-to-PRD through it yet. If the lint surfaces nothing real, dropping the pre-write gate was premature; if the OST chain proves friction-heavy in use, the chain is overbuilt for a pre-PMF product.
- **Multi-host portability (Claude / Vibe / Codex).** One skill source, three runtimes. Real ongoing maintenance tax (portability-v2 headers, Vibe subshell handling) for a pre-PMF tool.
- **A GUI above the CLI unlocks adoption.** The macOS viewer is the test: if form factor — not value — is the adoption blocker, a designed read-only surface should retain users the CLI doesn't. The viewer exists; the retention measurement does not.
- **Engine + recipes keeps the surface small.** Bet: expressing every skill as a thin recipe over three primitives makes the system cheaper to maintain and extend (add capability by writing a prompt, not plumbing). Risk: two read primitives (query + retrieval) instead of one is a small seam left deliberately.
