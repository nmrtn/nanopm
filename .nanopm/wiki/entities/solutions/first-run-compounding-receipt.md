## Pitch
End the first run with a generated receipt — a `nanopm_wiki_diff` shared primitive that emits "what this run added to your memory that you didn't have before" (entities created, links formed, claims evidenced) AND names the 2–3 concrete things the NEXT run can now do because the wiki exists ("next time, /pm-roadmap will read these 4 opportunities you just surfaced"). It sells the deferred payoff up front instead of asking the user to take it on faith — the one artifact ChatGPT structurally cannot produce. One primitive, every skill inherits it.

## Riskiest assumption
That a receipt of net-new wiki state reads as VALUE rather than bookkeeping noise — and that naming the future payoff actually changes return behavior rather than the barrier being that the 2nd run isn't worth it even when understood.

## Cheapest test
Hand-write the receipt output for one real first run and show it to 3–5 cohort users: does it produce an "oh, I couldn't get that from ChatGPT" reaction and a "yes, that would bring me back"? Decide before writing the primitive.

## Dissent/tension note
Eng: most durable option (one primitive, all skills benefit) but leans on a stable wiki schema mid-proof-quarter. Business tension: treats a value problem as a messaging problem — a prettier receipt is lipstick, and may burn trust by over-promising, if the 2nd run genuinely isn't worth it.
