## Problem Statement

A viewer user (Dani) has no way to know her installed nanopm skill pack is out of date, and no way to update it without leaving the app. nanopm ships often (four releases in the recent window; local is v0.11.0 while the skill pack a user installed weeks ago may be several versions behind), and the *only* update paths today are terminal-bound: re-run `curl … | bash setup`, or run `/pm-upgrade` from a CLI agent session. The non-terminal-native persona the viewer exists for can't do either — so she silently runs stale skills (missing adversarial-gate fixes, new Define skills, schema changes the next skill expects) with no signal anything is wrong. The CLI already detects this exact condition (`nanopm_update_check` emits `UPGRADE_AVAILABLE {local} {remote}`); the viewer simply doesn't surface it.

---

## User Stories

- As a viewer user (Dani), I want the app to tell me when a newer nanopm is available, so that I don't unknowingly run a stale skill pack.
- As a viewer user, I want to update the skills with one click from inside the app, so that I never have to open a terminal to stay current.
- As a returning user, I want confirmation that I'm now on the latest version, so that I trust the update actually happened.

## Scope

### In scope (v1)
- Background version check on viewer launch: read local `~/.nanopm/VERSION`, fetch remote `VERSION` from GitHub, compare with the same semver logic the CLI uses, respect the existing 24h cache.
- A passive, dismissible banner/block ("nanopm v{remote} available — you have v{local}") shown when remote > local. Claude-Code-style: unobtrusive, persistent until acted on or dismissed.
- An "Update now" button that runs the existing skills upgrade (re-runs the installed `setup`, the same operation `/pm-upgrade` performs).
- Silent execution: spinner while updating, then a success confirmation ("Updated to v{remote}") and the banner clears. (Per UX decision: no console shown.)
- Failure handling: if the update fails (no network, write error), show a plain error with a one-line fallback ("run `curl … | bash setup` in a terminal").
- **Maintainer guard (dev-install detection):** if the install was done from a local clone, the "Update now" button does **not** run the curl upgrade — it blocks and warns instead ("Dev install detected ({clone path}) — update from the terminal so you don't overwrite local work"). This is keyed on an automatic provenance signal, not a manual toggle, so it protects Guillaume & Nicolas without affecting end users (whose install is `remote`).

### Out of scope (v1)
- **Updating the viewer app binary itself** — revisit when the viewer has notarized releases + Sparkle; explicitly deferred (confirmed with user).
- **Visible upgrade console / terminal output** — deferred; a "view details" disclosure could come later if debugging demand appears.
- **Auto-update without a click** (the CLI's `auto_upgrade=true` path) — v1 is one-click only; honor the config if already set, but don't add a GUI toggle yet.
- **Snooze / backoff UI** — the banner is simply dismissible for the session; no escalating reminder schedule in v1.
- **Per-project vs global version awareness** — v1 targets the single global install at `~/.nanopm/`.

## Requirements

### Functional requirements
1. On launch, the viewer determines whether a newer nanopm exists by comparing local `~/.nanopm/VERSION` against the remote `VERSION`, reusing the CLI's semver comparison and 24h cache semantics (do not reimplement the comparison rules divergently — match `nanopm_update_check`).
2. When (and only when) remote is strictly newer, render the update banner naming both versions.
3. "Update now" executes the upgrade by shelling out to `curl -fsSL https://raw.githubusercontent.com/nmrtn/nanopm/main/setup | bash` (the same command `/pm-upgrade` uses for non-repo installs — always fetches latest from `main`, reinstalls every detected host). Surfaces only a spinner, then a result. Success/failure is read from the process **exit code** (0 = success); `setup` runs `set -euo pipefail` and exits non-zero on any error.
4. On success: update is reflected in `~/.nanopm/VERSION`, the banner clears, and a confirmation appears.
5. On failure: a non-technical error with the terminal fallback command; the banner remains.
6. The banner is dismissible for the current session without updating.
7. `setup` writes an install-provenance marker to `~/.nanopm/install-source` at the same point it writes `~/.nanopm/VERSION` — the local repo path when installed from a clone (`_SOURCE=local`), or `remote` when installed via curl. (One-line addition to `setup`; the values already exist as `_REPO_ROOT` / `_SOURCE`.)
8. On "Update now", the viewer reads `~/.nanopm/install-source`: if it is a clone path (dev install), block the curl upgrade and show the maintainer warning naming that path; otherwise (`remote` or marker absent) proceed with the silent curl update. Absent marker defaults to `remote` so pre-existing installs aren't falsely flagged as dev.

### Non-functional requirements
- The version check must not block or visibly delay app launch (async; fail-silent on no network — no error if the check itself can't reach GitHub).
- Reuse the CLI's update-detection contract so CLI and viewer never disagree about whether an update exists.

## The One UX Decision

**How the viewer triggers the upgrade.** Option A: the viewer shells out to the user's installed `setup`/upgrade script directly (a `ShellRunner` call), owning the spinner→result flow itself. Option B: the viewer hands off to a `/pm-upgrade` *skill run* inside its existing run console (reusing `RunManager`). Tradeoff: B reuses the most code and keeps one upgrade path, but it surfaces a run session (console-like), which contradicts the chosen "silent + result" UX and drags in agent-runtime dependency for what is a pure shell operation. A is a few lines of bespoke shell-out but delivers the silent, GUI-native experience the user asked for and works even when no agent session is active. **Recommend A**, calling the same `setup` operation `/pm-upgrade` wraps — so behavior matches the CLI without routing through an agent run.

## Resolved decisions (were open questions)

| Question | Resolution (from code) |
|----------|------------------------|
| Does `setup` run non-interactively (no prompts) when invoked headless? | **Yes, unconditionally.** `setup` has zero stdin reads / prompts — it's designed to be piped from curl. `set -euo pipefail` + `fail()` means it exits non-zero on any error, so the viewer gets a reliable success signal from the exit code. The silent spinner cannot hang on a prompt. |
| Re-run local `setup`, or fetch from GitHub? | **Fetch from GitHub via curl.** v1 always runs `curl -fsSL …/nmrtn/nanopm/main/setup \| bash` — universal (no local repo assumed), always pulls `main`, reinstalls all detected hosts. **No local-repo detection in v1** (the `git pull && bash setup` path is only relevant to maintainers working in a clone, not the viewer's target user). |

**Implementation notes:** After a successful exit, the viewer re-reads `~/.nanopm/VERSION` (written by `setup` at the end) to confirm the new version and clear the banner. `setup` also clears `~/.nanopm/last-update-check`, so the next launch check is fresh. Requires `bash` + `curl` on PATH and network access — both consistent with the version check, which already fetches the remote `VERSION`.

## Dependencies

- `bash` + `curl` on PATH (both standard on macOS) to run the upgrade command.
- `lib/nanopm.sh` `nanopm_update_check` / semver logic as the reference contract for "is there an update" (the path convention is load-bearing — keep CLI and viewer in agreement).
- Network access to `raw.githubusercontent.com/nmrtn/nanopm/main/` for both the remote `VERSION` (check) and `setup` + skill files (upgrade).
- A one-line change to the CLI `setup` script to write `~/.nanopm/install-source` (provenance marker for the maintainer guard) — small cross-surface change, ships alongside the viewer work.
