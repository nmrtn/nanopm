# PRD: Wiki search engine (DEFERRED)
Generated by /pm-prd on 2026-06-23
Project: nanopm
Status: DEFERRED — LATER (split out of memory-wiki-redesign v1)

## Problem
At current scale (tens of pages), an agent can find any wiki page by reading `wiki/index.md` — the always-loaded catalog — and following the named link. As the wiki compounds past that scale, scanning the index stops being reliable retrieval: the catalog itself grows, link names blur, and the agent starts missing pages that are relevant to a question. At that point nanopm needs proper retrieval (semantic + keyword) over `wiki/*.md`. Markdown + git remains the single source of truth; any search index is a disposable, rebuildable artifact derived from it — never load-bearing.

## Scope (when promoted to NOW)
### In scope
- Local, on-device search over `wiki/*.md` with no server/daemon and no hosted dependency. Preferred path: [qmd](https://github.com/karpathy) hybrid search (BM25 + vector embeddings + LLM rerank) invoked as a CLI the agent shells out to; fallback path: a small home-grown script (BM25 over the markdown corpus) if qmd is unavailable on a host.
- An MCP server wrapper so hosts that support it (Claude/Vibe/Codex) can call wiki search as a native tool, with graceful degradation to the CLI shell-out where MCP is absent.
- Index build/refresh from `wiki/*.md` on demand (and as a step the lint/sleep agent can trigger), so the index always reconstructs from markdown.
### Out of scope
- Anything that makes search load-bearing: the index is never the source of truth, is `.gitignore`d, and the wiki must remain fully usable (read + ingest + lint) with the index deleted. Markdown + git stays the only canonical store.
- Replacing the always-loaded `index.md` catalog — search augments retrieval for large wikis; it does not become the startup load path (startup stays deterministic bash, never a subagent or search call).

## Revisit trigger (what promotes this from LATER to NOW)
Promote when ANY holds: the wiki exceeds ~150 pages, OR `wiki/index.md` exceeds ~50 KB (no longer cheap to load + scan every run), OR a maintainer observes the agent measurably missing relevant pages when answering via the index alone (≥2 such misses across 10 runs, mirroring the parent PRD's falsification cadence).

## Success criteria
- Agent retrieves the correct page(s) for a natural-language query at ≥90% top-3 hit rate on a maintainer-curated probe set, beating index-only retrieval at the same wiki size.
- Deleting the entire index and rebuilding from `wiki/*.md` yields identical retrieval behaviour and zero content loss — proving the index stays disposable.
