## Problem Statement

The viewer's Opportunities page can render the opportunity DB but cannot change it. A PM who spots a missing user problem while reading the list — or who wants the agent to surface more — has to leave the GUI, open a terminal, and run `/pm-opportunities` by hand. For the persona the viewer exists to serve (Designer-founder Dani, non-terminal-native) that is a dead end: the killer-demo DB is read-only to the exact user who can't drop to a shell. Even for terminal-native Theo it is a context switch out of the surface he was just reading, breaking the "planning compounds in one place" job-to-be-done — today his workaround is "bounce four tools every planning cycle" and the read-only viewer adds a fifth bounce instead of removing one (PERSONAS.md).

The cost is concrete and strategic: the Discovery Opportunity DB is the NOW #1 bet and the recruitment hook for the Q3 proof quarter (PLAN-SUMMARY). A demo where you can browse opportunities but must use the terminal to add one is a broken demo for the audience the prototype is meant to convert.

*No DATA.md or FEEDBACK.md exists yet (pre-PMF, 0 external users measured), so no quantified drop-off or verbatim quote is cited here — the problem is grounded in PERSONAS.md and the current plan, not in usage metrics.*

## User Stories

- As **Dani (non-terminal-native, the viewer's target persona)**, I want to add an opportunity I just thought of by typing it into the Opportunities page, so that I can contribute to the DB without ever opening a terminal.
- As **Theo (terminal-native solo founder)**, I want to ask the agent to generate a few more opportunities — globally or for one theme — from a button on the page I'm already reading, so that I stay in one surface instead of bouncing to the shell.
- As either PM, I want the run I launched to show progress and then refresh the list in place, so that I see the new opportunity appear without reloading or hunting through files.

## Scope

### In scope (v1)
- A launch control in the Opportunities overview header: a `+ Run ▾` menu (matches the approved design).
- **Mode A — "Add one from text":** opens a sheet/popover with a multiline field for the user problem; on Run, launches `/pm-opportunities` with the typed text as input, **routed through the dedup agent before it writes**. Reuses the existing `SkillRunButton` popover pattern.
- **Mode B — "Generate N more":** submenu **Global** or **By theme…**; the theme list is derived from existing opportunities' `theme:` frontmatter (live values today: *Adoption & form factor, Continuity & retention, Onboarding & guidance, Trust & evidence*). On Run, launches `/pm-opportunities` in a new additive generate mode; **every drafted candidate passes through the dedup agent** before it lands.
- **The dedup agent (shared, reusable) — see Requirement 8:** a standalone subagent that decides, for each candidate opportunity, whether it is *new*, a *duplicate*, or should *merge into* an existing one. v1 builds it as a first-class reusable agent (NOT inline logic in `generate`) precisely because it sits in front of **every** opportunity write path — manual add, generate, and the future signal-extraction pipeline.
- **Skill extension:** teach `/pm-opportunities` to (a) parse a structured hint in the run context (`add: …` / `generate: …`) and route directly instead of auto-detecting, and (b) support an additive **generate** mode that drafts N new opportunities (reusing bootstrap's per-theme fan-out), runs each candidate through the dedup agent, writes survivors with `provenance: nano-hypothesis`, appends to `LOG.md`, and never overwrites existing files. Backward compatible: with no hint, current auto-detect behavior is unchanged.
- **Empty-DB case:** if `.nanopm/opportunities/SCHEMA.md` doesn't exist, the button runs the existing **bootstrap** path.
- In-flight handling: disable the button while a run for this artifact is active (reuse `RunManager.isActive` on the tracking path); refresh the list on completion (observe `completionTick` → `store.refresh()`).

### Out of scope (v1) — explicitly deferred
- **Edit / delete an opportunity from the GUI** — revisit if a cohort user names read-only editing as a blocker. Editing stays in the `.md` files / terminal skill.
- **Review-before-write gate for generated opportunities** — revisit if generated quality demonstrably pollutes a demo DB. v1 writes directly as `nano-hypothesis` and the PM reviews *after* in the list.
- **Re-ranking / re-prioritisation button** — revisit later; coarse priority stays skill-assigned at write time.
- **Model picker / exact-N numeric control** — revisit on a named request. v1 uses a sensible default N (3–5) and the default run model.
- **The signal-extraction pipeline (user-research / sales call transcripts → opportunities)** — out of v1, but it is the named **second consumer** that motivates building dedup as a reusable agent now. v1 ships the dedup agent with a clean, contract-stable interface; wiring transcripts into it is a future Discover-phase feature with its own PRD. The point of building the agent here is that *that* feature reuses it unchanged.

## Requirements

### Functional requirements
1. Add a `+ Run ▾` menu to `OpportunitiesOverviewView`'s header (`OpportunitiesUI.swift`), visible whenever the Opportunities page is shown.
2. **Mode A:** menu item "Add one from text…" opens a sheet with a multiline `TextField`; a Run button calls `RunManager.launch(opportunitiesDoc, in: projectPath, userContext: "add: <text>")`. The skill runs the typed problem through the **dedup agent (req 8)** before writing. Cancel dismisses without launching.
3. **Mode B:** menu item "Generate N more" with subitems "Global" and one entry per live theme under "By theme…". Selecting launches `RunManager.launch(..., userContext: "generate: <N> [for theme <theme>]")`.
4. Extend `/pm-opportunities` (`SKILL.md`) to parse `add:` / `generate:` hints from the run context and route accordingly; implement the additive `generate` mode (per-theme or global fan-out, **each candidate passed through the dedup agent (req 8)**, write `nano-hypothesis` + `LOG.md`, no overwrite).
5. Update the opportunities `headlessArgs` in `SkillCatalog.swift` so the skill is told to honor the hint and fall back to auto-detect when none is present.
6. If no `SCHEMA.md`, the menu collapses to a single "Bootstrap opportunities" action running the existing bootstrap path.
7. Disable the launch control while a run for the opportunities artifact is active; on completion, trigger a store refresh so new files appear without manual reload.
8. **The dedup agent (shared component).** A standalone subagent invoked by `/pm-opportunities` (e.g. `pm-opportunity-dedup`), built reusable — NOT inlined in `generate`:
   - **Input:** one or more candidate opportunities (title + problem text + optional verbatim quote + provenance) **and** the current `.nanopm/opportunities/` DB (existing files' titles + problem summaries — read from disk, since opportunities are an LLM-wiki).
   - **Output (per candidate):** a verdict — `new` | `duplicate-of:<slug>` | `merge-into:<slug>` — plus a confidence (0–10) and a one-line rationale naming the matched slug.
   - **On `merge-into`:** append the candidate's phrasing / quote / evidence to the existing opportunity's file and write a `LOG.md` entry; do **not** create a new file. On `new`: write a new file. Never silently overwrite.
   - **Context-aware action policy (the verdict→action mapping):** *interactive single add* (Mode A) asks the user on a high-confidence match (merge into X / keep as new / cancel); *batch* contexts (Mode B generate, and the future signal-extraction pipeline) decide non-interactively — auto-merge above a high-confidence bar, else create-new and tag `related-to:<slug>` — and report a one-line merge/skip summary at the end of the run. See "The One UX Decision."
   - **Reusability is a hard requirement:** the contract above must be stable enough that a future transcript→opportunity extractor calls the same agent with no interface change. If a design choice here would force that future caller to fork the agent, it's the wrong choice.

### Non-functional requirements
- No new run-execution, concurrency, or process-management code *in the viewer* — reuse `RunManager`/`ShellRunner` as-is. Mode B's per-theme fan-out happens **inside the single skill run** (the skill spawns its own subagents, exactly as bootstrap already does); the viewer launches one run per click and the existing one-run-per-artifact gate holds. A multi-theme "Generate N more" is a skill-internal loop, **not** parallel viewer runs.
- Skill change must be backward compatible (terminal `/pm-opportunities` with no hint behaves exactly as today).
- Generated opportunities must be visually distinguishable in the list via the existing provenance badge — a low-quality batch must never look like evidence-backed signal.

## The One UX Decision

**When the dedup agent flags a candidate as a likely duplicate, does the run auto-merge it or stop and ask the user?**

The dedup agent (req 8) is what makes direct-write safe — without it, "Generate N more" and repeated manual adds would silt the killer-demo DB with near-duplicates, and *that* is the failure mode that makes the demo worse instead of better. But a dedup agent that guesses wrong is its own failure: auto-merging two *distinct* problems destroys signal, while asking on every candidate turns a 5-item generate run into a 5-prompt interrogation.

- **Option A (chosen) — context-aware policy:** *interactive single add* (Mode A) asks the user on a high-confidence match (merge into X / keep as new / cancel); *batch* runs (generate-N, future signal extraction) decide non-interactively — auto-merge only above a high-confidence bar, otherwise create-new and tag `related-to:<slug>` — then report what merged at the end.
- **Option B (rejected) — one policy everywhere:** *always-ask* kills batch ergonomics and can't work for non-interactive signal extraction at all; *always-auto* silently merges distinct problems in the one demo where trust matters most.

**Tradeoff:** the context-aware policy is more to build (two paths) and needs a tuned confidence threshold (see Open questions) — but it's the only option that serves both the single-add UX *and* the batch/transcript path the agent is explicitly being built to outlive this feature for. The residual risk is a wrong auto-merge in batch mode; the guard rail (also a hard requirement) is that **every merge is logged to `LOG.md` with its source candidate**, so it's auditable and revertable in seconds. Generated/added rows still land as `provenance: nano-hypothesis` with an unmissable badge — direct-write stays, but dedup is what keeps it honest.
