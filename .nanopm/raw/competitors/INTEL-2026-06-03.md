## Summary

Three of four competitors shipped material change since May 21. Two emergent patterns matter:

1. **Plugin-based install is now the convergent distribution model** across gstack, mycelium, and prawduct. All three moved from file-sync / vendored copies to brownfield-safe plugins. nanopm uses a setup script that copies into `~/.{host}/skills` — same effective outcome, but the language is "skill pack" instead of "plugin." Worth deciding whether to adopt the convergent vocabulary.

2. **Mycelium has crystallized its positioning with a sharp tagline:** *"AI has made building cheap. It hasn't made deciding cheap."* This is the cleanest competitive headline in the space and is aimed at the same audience nanopm targets. Mycelium also fired an explicit shot at "49 skills dumped simultaneously" framing — clearly pointed at deanpeters' library and any tool that conflates breadth with value.

**Action:** Read mycelium's new README in full before next strategy review — their positioning is now substantially sharper than nanopm's. Decide whether nanopm should lean into a similar one-line thesis or differentiate on a different axis (e.g., "the PM half that hands off cleanly to 5 delivery layers" vs mycelium's discovery-rigor framing).

## Strategic implications

### 1. The discovery-rigor lane has a clear leader now
Mycelium's "AI has made building cheap. It hasn't made deciding cheap." is the sharpest one-line positioning in this space. They've combined it with explicit anti-positioning (rejecting universality, naming the failure modes they DON'T address). nanopm's "PM half" framing is honest but fuzzier. **Either lean into a similarly sharp tagline of our own, or accept that mycelium owns the "discovery rigor" lane and nanopm differentiates on something else** (most natural: the symmetric handoff story — "the PM half that plugs into your delivery stack — Linear, GitHub, OpenSpec, gstack, or human").

### 2. Plugin distribution is now standard table-stakes
gstack, mycelium, and prawduct all use plugin-based brownfield-safe installs. nanopm uses a setup script that copies into `~/.{host}/skills` — same effective outcome but different vocabulary. **Worth either:**
- Adopting "plugin" language explicitly in the README + install copy, OR
- Differentiating by explaining why the script approach is better (e.g. multi-host symmetric install in one command — which is genuinely something the plugin approach doesn't do cleanly).

### 3. gstack vs nanopm on telemetry: explicit strategic divergence
- **nanopm v0.6.0** removed telemetry as pre-PMF over-engineering. Honest, but loses observability of what users actually use.
- **gstack v1.55** doubled down on telemetry but made the consent screen brutally explicit + pinned the guarantees with regression tests. Maintains observability without sacrificing trust.

Both are defensible. The divergence is real and worth noting in any "how it compares" content.

### 4. mycelium's "anti-pattern" naming is a moat-builder
They're tagging failure modes (AP#1 through AP#8+) and shipping mechanisms that catch each one structurally. As the catalog grows, the framework becomes a defensible body of named, tested protections. nanopm could follow the same pattern with its decision types and gates — explicit names for the failures each gate catches.

### 5. Velocity asymmetry
gstack and mycelium are both shipping ~1 minor version per day. nanopm has shipped v0.6.0 → v0.6.4 in roughly the same period — also fast. prawduct and deanpeters are slower. **The space is moving fast. Re-running this intel weekly (not bi-weekly) would catch material change earlier.**
