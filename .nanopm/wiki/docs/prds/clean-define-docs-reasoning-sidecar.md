# PRD: Clean Define documents with a separate reasoning sidecar

## Problem Statement

When a Define skill (`/pm-vision-mission`, `/pm-business-model`, `/pm-org`, `/pm-product`, `/pm-personas`) generates its artifact, it interleaves the model's *reasoning* — `Confidence: Evidenced — source | Assumed — basis` tags and rationale prose explaining why each decision was made — directly inside the document. The artifact is therefore a working document, not a shareable one. Today the founder's workaround is to hand-strip that meta before showing `VISION-MISSION.md` or `PRODUCT.md` to anyone outside the repo — a manual cleanup pass on every share, repeated for every regeneration. The cost: the artifacts nanopm produces, which are meant to be the deliverable, are never directly shareable, undercutting the "planning that compounds and is usable" promise to the primary persona (Terminal-native Theo, who wants outputs he can hand to a teammate, not just to himself).

Note: planning against a `complete` product concept (PRODUCT.md).

---

## User Stories

- As a solo founder using nanopm, I want each Define document to come out as a clean, claim-only artifact, so that I can share it with my team or stakeholders without a manual editing pass.
- As that same founder reviewing the output, I want the model's reasoning — what's evidenced vs. assumed, the sources, the "why this call" — preserved in a companion document I can open in the viewer next to the clean doc, so that I can audit the thinking without it polluting the deliverable.

If that holds, the auto-generated clean doc is not actually share-ready and the split failed to solve the problem.

## Scope

### In scope (v1)
- All 5 Define skills (`pm-vision-mission`, `pm-business-model`, `pm-org`, `pm-product`, `pm-personas`) produce two outputs per run: a **clean** `.nanopm/<DOC>.md` (claims only — every `Confidence:` / `Evidenced` / `Assumed` / source / rationale removed) and a **reasoning sidecar** at `.nanopm/reasoning/<DOC>.md`.
- The reasoning sidecar mirrors the clean doc's section headings, so a reader can find the rationale for any section by matching heading. Per section it carries: the Evidenced/Assumed call, the source, and the short "why this decision."
- Viewer pairs the two: when viewing a clean Define doc, the reasoning sidecar is reachable from that document's detail view (not as a stray sibling entry in the sidebar).
- A shared path convention so skills and viewer agree on where the sidecar lives (e.g. a `nanopm_reasoning_path` helper in `lib/nanopm.sh`).

### Out of scope (v1)
- **Non-Define docs** (Strategy, Roadmap, PRD, Feedback, Discovery, etc.) — keep current format. Revisit once the Define split is proven.
- **Retrofitting existing artifacts** — no migration of already-generated `.nanopm/` docs; only new generations produce the clean+sidecar pair. Revisit if founders want a one-shot backfill.
- **Editing/annotating the reasoning in the viewer** — read-only display only. Revisit if the viewer gains write capabilities.

## Requirements

### Functional requirements
1. Each Define skill's document template is rewritten so the clean artifact contains only the substantive claims — no `Confidence:` line, no `Evidenced/Assumed` tag, no inline "why."
2. Each Define skill gains a write step that emits `.nanopm/reasoning/<DOC>.md`, structured to mirror the clean doc's headings, carrying the Evidenced/Assumed tags, sources, and rationale that were previously inline.
3. The `.nanopm/reasoning/` directory is created on demand (like `.nanopm/prds/`) and is gitignored with the rest of `.nanopm/`.
4. The viewer's scanner recognizes `reasoning/<DOC>.md` as the companion of `.nanopm/<DOC>.md`, excludes it from the top-level artifact list, and exposes it from the clean doc's detail view (e.g. a "Reasoning" toggle/tab).
5. The viewer renders the sidecar as markdown using the existing markdown component — no new rendering path.

### Non-functional requirements
- The `CONTEXT-SUMMARY.md` regeneration subagent reads the **clean docs only** — it must NOT read the reasoning sidecars (founder decision, 2026-06-12). The brief therefore works from claims; it marks `(assumed)` from its own judgment rather than from the sidecar's Evidenced/Assumed tags.

---

## The One UX Decision

How the viewer surfaces the reasoning sidecar.

- **Option A — Paired toggle in the document detail view:** viewing `VISION-MISSION.md` shows a "Reasoning" toggle/tab that swaps to (or splits to) `reasoning/VISION-MISSION.md`. The sidecar never appears as its own sidebar row. This matches the user's framing ("consultable lorsqu'on regarde le document") and keeps the artifact list clean.
- **Option B — Separate sidebar entry:** the reasoning file shows as its own row (the scanner already maps it to the Define phase by filename prefix, so this is near-zero work). Cheaper, but clutters the list and decouples the reasoning from the doc it explains.

Tradeoff: Option A costs viewer work (pairing logic + a toggle in `ArtifactDetailView`) but delivers the intended experience; Option B is almost free but is the thing the user is implicitly asking us *not* to do. **Recommendation: Option A.**
