# PRD: Karpathy-faithful memory engine (engine + recipes)

Generated by /pm-prd on 2026-06-25
Project: nanopm
Status: DRAFT

---

## Problem Statement

nanopm's memory wiki has drifted from the simple model that inspired it (Karpathy's LLM-wiki) and over-built in three places. We added a **confidence gate + `_review` approval queue** — pre-write governance the source model doesn't have, layered onto every entity ingest. We built the **mechanical half of lint** (frontmatter + broken-link checks) and left the **judgment half** (does this page contradict another? is there a gap?) written but unwired — the less useful half is the one that runs. And we have **no first-class Query operation**, which the source model treats as a core pillar. The cost lands on the two maintainers — more code to keep alive, governance whose value isn't proven, and a quality check (lint) that inspects formatting instead of truth — and we're about to harden this heavier shape by routing more skills onto the gate, right before we'd simplify it. Surfaced by Nico's deep review of PR #121.

The constraint that makes this worth doing now: the faithful version is *simpler* than what we built, and #121 (wiki = single source of truth) is the foundation it sits on — so the moment to trim is right after #121 lands, before more skills are wired to the gate.

---

## User Stories

- As a **nanopm maintainer**, I want the memory core to be the three Karpathy operations (ingest / query / lint) so I add capability by writing prompts, not by extending governance plumbing.
- As a **nanopm maintainer**, I want skills to be thin recipes over those primitives (query → reasoning → ingest) so wiring a new skill is composition, not bespoke read/write code I have to keep migrating.
- As a **nanopm user**, I want lint to flag real contradictions and gaps — not just missing frontmatter — so my wiki's quality signal is worth reading instead of permanently noisy.

## Falsification

**The central bet:** removing the pre-write gate is safe *because* a wired judgment lint catches contradictions after the fact (Karpathy's "write freely, lint surfaces, human curates"). That bet is load-bearing — so it's the one to falsify.

If, across our three live wikis (**nanopm, Forto, runningcoach**), the wired judgment lint surfaces **zero** real contradictions or gaps in its **first 3 weeks** of running that the old mechanical lint wouldn't already have caught — i.e. it adds no truth-signal — then the "the lint is a good-enough safety net to drop the gate" bet is wrong, and we pause before un-wiring the gate from the remaining skills.

## Scope

### In scope (v1)
- Skiplist the opportunities housekeeping files (`INDEX.md` / `LOG.md` / `SCHEMA.md`) in `entity_pages()` (lint) and `collect_pages()` (reindex) — reuse the dedup agent's existing skiplist. *(This is review item #2; it's the only real gate↔migration coupling, so it lands here.)*
- Wire the judgment lint (`nanopm_lint_prompt`) as the active lint pass; keep the mechanical checks as a cheap pre-filter.
- Remove the confidence gate + `_review` queue; un-wire the ~5 signal skills that call it.
- Add a `query` primitive to `lib/nanopm.sh` (read relevant pages → synthesize → file the answer back) and route every reading skill through it.
- **Re-express all skills as recipes** (query → reasoning → ingest) — every `pm-*` skill, not a proof subset. No skill keeps bespoke read/write plumbing or confidence-gate calls.

### Out of scope (v1)
- Generic `/pm-ask` user command — revisit once the query primitive has earned its keep internally.
- The upgrade flat-path fallback for per-doc reads (review item #1) — explicitly punted; clean migration-on-upgrade is folded into the trimmed-core schema instead.
- Touching the adversarial PM gates or the 3-layer storage.

## Requirements

### Functional requirements
1. The opportunities `INDEX/LOG/SCHEMA` files are excluded from lint and reindex (no permanent "missing-frontmatter" errors; `LOG.md` never enters the always-loaded index).
2. The lint agent runs the judgment prompt and reports contradictions/gaps, not only frontmatter/link errors.
3. The confidence gate and `_review` queue are removed; no skill blocks a write on human approval. Contradictions are surfaced by lint, not gated before write.
4. A `query` primitive exists in the shared lib and is called by every skill that reads upstream artifacts.
5. Every skill is re-expressed in recipe form (query → reasoning → ingest); no skill retains bespoke read/write plumbing or confidence-gate calls.

### Non-functional requirements
- A one-page **trimmed-core spec** (schema + the 3 prompts: ingest / query / lint + migration-on-upgrade) is co-signed with Nico before code changes begin.

---

## The One UX Decision

With the gate gone, **how do contradictions reach the human?**

- **Option A — log-only (faithful, quiet):** lint writes contradictions to `log.md`; the user notices on their own. Simplest, truest to Karpathy, but easy to ignore.
- **Option B — surface at preamble top (loud):** lint findings are shown at the start of each run, like today's `_review` surfacing. Safer/visible, but re-creates some of the "review surface" feel we're cutting.

Tradeoff: fidelity + simplicity (A) vs. safety + visibility (B). This decision *is* the load-bearing risk from the Falsification section — pick it deliberately, don't default into it.

## Sequence (how it rolls out)

1. Merge #121 + the #2/#3 fixes (foundation; not part of this PRD's build).
2. Co-sign the trimmed-core spec (schema + ingest/query/lint prompts + migration-on-upgrade).
3. Wire the judgment lint **first** — it's the safety net that makes step 4 safe.
4. Remove the confidence gate + `_review`; un-wire the signal skills.
5. **First wave:** add the query primitive and re-express 2-3 skills as recipes (query → reasoning → ingest) on the now gate-free path.
6. **Observe** the wired judgment lint across our three wikis over the falsification window (~3 weeks) — is it catching real contradictions the mechanical pass missed? (This is the bet from the Falsification section.)
7. **Convert the rest:** re-express the remaining skills as recipes. All within this PRD — the checkpoint at step 6 is a gate *inside* it, not a separate follow-up.

Cut-then-build, with the lint landing before the gate's safety role is dropped, so no quality hole opens mid-transition. The first wave (steps 5-6) de-risks the load-bearing bet cheaply before the bulk conversion (step 7) commits to the gate-free world.

## Reviewer notes

Consider promoting a thresholded version of that into the Success Criteria.

Mitigation (chosen): stage the rollout *within* this PRD — a first wave (gate removal + lint wiring + 2-3 skills) observed over the falsification window, then the remaining skills — so it stays one PRD with no separate follow-up, but the bulk waits for the cheap signal.
