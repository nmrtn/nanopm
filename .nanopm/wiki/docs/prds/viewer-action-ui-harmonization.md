# PRD: Viewer action UI harmonization (Run / Refresh / Reasoning)

## Problem Statement

In the SwiftUI viewer, the three actions a user takes most often — **launch a Run**, **refresh content from disk**, and **view the reasoning sidecar** — are each rendered three different ways depending on which screen you're on. Run is a plain text-only button on the phase overview (`PhaseOverviewView.swift:184`) but a `play.circle` menu in the competitor view (`CompetitorDetailView.swift:145`). Reasoning is a segmented picker plus a window button in the artifact detail (`ProjectView.swift:432-454`) but a `brain` toggle in the competitor view (`CompetitorDetailView.swift:152`). Refresh is an unlabelled icon tucked into the sidebar footer (`ProjectView.swift:168-175`). There is **no shared button component** — every control is hand-styled with ad-hoc modifiers, so the same action looks and sits differently on each screen. For the viewer's intended user (Designer-founder Dani, a non-terminal-native who judges the tool on whether it reads as "a polished demo over nothing"), this inconsistency reads as unfinished and makes the primary actions harder to find than they should be.

---

## User Stories

- As a non-terminal founder using the viewer, I want the Run, Refresh, and Reasoning controls to look and behave the same on every screen, so that I can act on a document without re-learning the UI each time I navigate.
- As a viewer user looking at any document, I want each action to carry a clear icon **and** a label, so that I can tell at a glance what each button does without hovering for tooltip text.

## Falsification

If, within **14 days** of shipping the harmonized action bar, **2 of the 3** people dogfooding the viewer (Nicolas, Guillaume, or a recruited viewer-cohort tester) still have to ask aloud or over DM "where do I launch a Run / open the reasoning" from a document view — **or** a follow-up commit re-introduces a divergent control type for one of these three actions on any single screen — then the harmonization failed to make the actions self-evident and consistent, and the bet (that unifying these controls improves the viewer's usability and "finished" credibility) is wrong.

## Scope

### In scope (v1)
- Unify **Run**, **Refresh**, and **Reasoning** into three discrete, well-separated buttons, each with an SF Symbol icon **and** a text label, sharing one consistent treatment.
- Apply the harmonized treatment across all three call sites: `PhaseOverviewView` (Run), `ProjectView`/`ArtifactDetailView` (Refresh + Reasoning), and `CompetitorDetailView` (Run + Reasoning).
- Preserve Run's dynamic states ("Run" / "Answer…" / "Running…" with SparkleView) inside the new harmonized button — same logic, new consistent form.
- Harmonize the styling of the Competitors Run **menu** (the 3-option Intel/Find/Full menu) and the reasoning-in-separate-window button so they read as part of the same family, reusing existing `np*` colors.

### Out of scope (v1)
- New theme, color system, or design tokens — reuse existing `np*` colors as-is. Revisit only if a broader viewer visual refresh is greenlit.

---

## Requirements

### Functional requirements
1. Introduce a single shared action-button view (e.g. `ActionButton`) taking an SF Symbol, a label, an action, an enabled/disabled state, and an optional "busy/state" variant; render icon + label with consistent spacing and `np*` styling.
2. Replace the Run control in `PhaseOverviewView.swift:184` with the shared button, keeping the popover-for-context and `runManager.launch(...)` behavior and the "Run / Answer… / Running…" states.
3. Replace the Refresh control in `ProjectView.swift:168-175` with the shared button — now carrying a visible label (e.g. "Refresh") alongside the `arrow.clockwise` icon — keeping `store.refresh()` and existing auto-refresh triggers.
4. Replace the Reasoning controls in `ArtifactDetailView` (`ProjectView.swift:432-454`) and `CompetitorDetailView.swift:152-161` so the toggle/open-reasoning action uses the shared button family with a `brain` icon + "Reasoning" label, consistent across both.
5. Bring the Competitors Run menu (`CompetitorDetailView.swift:145-150`) and the reasoning-in-separate-window button (`macwindow.on.rectangle`) into the same visual family (consistent icon/label/styling), without removing their distinct functionality.
6. Keep all existing `.help()` tooltip text on each control.

### Non-functional requirements
- No regression in existing behavior: launching, resuming (Answer…), refreshing, document/reasoning pane switching, and separate-window reasoning all continue to work.

## The One UX Decision

**Run on a skill row: keep the single "Run" button, or surface its variant actions the same way Competitors does?**

- **Option A — single button everywhere:** Run is always one button; the Competitors view's three-way menu (Intel / Find / Full) stays a special case behind that one button's affordance. Maximum consistency, but the Competitors menu remains the one place the pattern bends.
- **Option B — button that reveals variants when they exist:** the shared button optionally acts as a menu when an action has variants (Competitors), and as a plain button when it doesn't (every other skill). One component, two modes.

Tradeoff: Option A is simpler and reads as cleaner, but leaves Competitors visibly different. Option B unifies the *component* at the cost of a more complex shared button. Recommendation: **Option B** — it's the only choice that actually harmonizes the Competitors screen rather than exempting it, which is exactly where today's inconsistency is worst.
