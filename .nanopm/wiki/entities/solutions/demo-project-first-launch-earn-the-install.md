## Pitch
Reframe the viewer first-run as a guided demo project, not an install flow. The downloaded app opens into a pre-populated sample project (a finished nanopm pipeline on a real-looking SaaS) so the user sees a full Define→Discover→Plan→Build experience in 30 seconds — before any folder picker, CLI install, or host-pack decisions. A persistent "Use this on my project" CTA in the top bar is the moment we ask for the folder + CLI install. We earn the install by showing the value first.

## Riskiest assumption
That showing a canned demo of someone else's project creates enough desire to convert to "install on my own project" — rather than just admiring it for 30 seconds and closing the app like a screensaver.

## Cheapest test
Record a 90-second screen video of the demo-first flow, send to 5 designer-founders, ask: "After watching this, would you click 'Use on my project'? Why?" Look for ≥3 unprompted "yes, because…" answers.

## Dissent/tension note
Cross-lens — direct conflict with the opportunity guardrail "a frictionless install must still keep the viewer tethered to a real project." A demo-project mode is exactly the slip toward standalone-PM-SaaS the anti-persona watch warns about; once founders can browse without a project, some will stay there forever and we'll be tempted to serve them.
