## 1. Problem summary
nanopm's loop ends at breakdown — once a PRD is broken into tasks and handed off, there is nothing to help the founder ship the *communication* around what ships. The Photoroom-shaped post-ship comms layer is absent: an always-up-to-date changelog, a release note for the feature (user benefits plus the "why"), and an anticipated FAQ for stakeholders that support can then turn into help-center pages.

## 2. Value to the user
### Job to be done
After breakdown, the founder wants nanopm to help him communicate the work outward — keep a changelog current, draft a release note that explains the user benefit and the rationale, and produce an anticipated stakeholder FAQ that feeds support content. His alternative today is writing all of that by hand outside nanopm, disconnected from the upstream opportunity/solution/PRD context that would make it accurate.

### Where we fall short
**The loop stops at breakdown — no comms artifacts**
There's no changelog, release note, or anticipated FAQ generated from the shipped work.
- "Il me manque quelque chose pour shipper derrière : un changelog toujours à jour, une release note pour ma feature... une FAQ anticipée pour les stakeholders, utilisée ensuite par le support pour générer les pages du centre de support." — self-interview, 2026-06-29 (Granola dc61e5c2)

## 3. Value to the company
Extends nanopm's reach across the full PM lifecycle (discover → plan → build → *communicate*), and these artifacts are uniquely well-positioned: they can draw on the same upstream opportunity/solution/PRD context the rest of the wiki holds, so the "why" is accurate by construction rather than re-derived by hand.
