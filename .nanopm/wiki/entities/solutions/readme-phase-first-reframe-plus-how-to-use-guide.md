## Pitch
Rewrite the public README around the four phases, not the skill list. Lead with "Define → Discover → Plan → Build — run `/pm-run`"; each phase gets a one-paragraph "what you get, what it costs, what unlocks next." Slash commands become an appendix. Add a new `docs/HOW-TO-USE.md` — the "maximize impact" guidelines document the founder asked for verbatim — covering when to run each phase, how the compounding state layer works, and what "using nanopm well" looks like across a real project. Ladders directly to The Bet's load-bearing "get guided on when and why to run each skill" and to Obj2 (a phase-oriented README reads as a product, not a toolbox — the GitHub-discovery conversion surface).

## Riskiest assumption
Terminal-native power users (Theo, Nico) will accept phase-first framing without feeling talked down to — the reframe doesn't cost us the power-user star base while making the surface legible to newcomers.

## Cheapest test
Draft the README rewrite + `HOW-TO-USE.md` as a draft PR, post it to the 3 non-friend engaged Obj2 users. Ship if 2/3 say "clearer" and 0/3 say "condescending." Cost: one afternoon.

## Dissent/tension note
Business: this is a docs change dressed as strategy; without also culling skills (Solution A: [[cull-the-roster-before-adding-guidance]]) the phase framing still has to defend a 21-to-24-skill roster, which strains credibility. Won't cost much alone — but may not move the needle either.
