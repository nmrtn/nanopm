# PRD: Artifact Quality Pass
Generated by /pm-prd on 2026-03-28
Project: nanopm
Status: DRAFT

---

## Problem Statement

Solo founders running the nanopm pipeline receive artifacts (AUDIT.md, OBJECTIVES.md, STRATEGY.md, ROADMAP.md, PRD) that are too generic and non-actionable. The outputs read as though they could apply to any product — vague sections that restate what the user already knows without pointing to a clear next step. The user finishes the pipeline, reads the artifacts, and changes nothing about what they were going to build anyway. The workaround is continuing to use ad-hoc ChatGPT prompts (the behavior nanopm is trying to replace), because at least those feel responsive even if they don't persist. The cost: zero behavioral change, zero retention loop, zero differentiation from the tool they already use.

---

## User Stories

- As a solo founder running `/pm-run`, I want every section of AUDIT.md to surface something I didn't already know about my product, so that the audit is worth reading rather than skimmable filler.
- As a solo founder reading the PRD, I want a clear, actionable next step in every section, so that I can hand it to myself (or a collaborator) and start building without a follow-up planning session.
- As a solo founder who completed the pipeline, I want the STRATEGY.md to name a specific, falsifiable bet — not a strategic platitude — so that I know what I'm testing against reality.
- As a solo founder using nanopm on a second product, I want the outputs to feel observably different from the first product's artifacts, so that the tool proves it's reading context rather than generating boilerplate.
- As a solo founder running the pipeline for the second time on the same product, I want the new artifacts to reflect what changed — not regenerate the same content — so that the context-compounding advantage is visible.

---

## Success Criteria

| Criteria | How Measured | Target |
|----------|-------------|--------|
| Section-level actionability | Pick any section from AUDIT.md or PRD.md, ask "what do I do next?" — is the answer derivable without follow-up questions? | Yes for every section reviewed across 3 sampled pipeline runs |
| Behavior change confirmation | User interview or async message: "the output changed what I built" | 2 of 3 pilot users confirm after running the polished pipeline |
| Output specificity | Manually review: does any sentence in AUDIT.md apply identically to a different product in the same category? | Zero interchangeable sentences per artifact |

**Anti-goals:** Success is NOT measured by artifact length, number of sections, or user satisfaction ratings alone. A shorter, more specific artifact outperforms a longer generic one.

---

## Scope

### In scope (v1)
- Prompt improvements to all 5 core skills: pm-audit, pm-objectives, pm-strategy, pm-roadmap, pm-prd
- Question sequencing improvements: if a question's answer is derivable from context, skip it and explain why
- Adversarial challenge refinement: ensure the challenge in pm-strategy names a specific falsifiable assumption, not a category of risk
- Output format tightening: remove or replace sections that consistently produce generic content across test runs
- Test methodology: run pipeline on 3 real products (nanopm + 2 others), compare before/after on the three criteria above

### Out of scope (v1)
- **New skills or new artifact types** — polish what exists; adding more surface area before quality is validated compounds the problem
- **Connector integrations (Linear, GitHub)** — these are a data quality improvement, not a prompt quality improvement; sequence after prompts are validated
- **Automated quality scoring** — manual review across 3 pipeline runs is sufficient for v1; automation is premature

---

## Requirements

### Functional requirements
1. Each skill must check if a question is answerable from prior context before asking it — if yes, populate the answer and state the source, rather than asking.
2. Every artifact section must end with (or contain) an explicit, imperative next action — not a recommendation, an action: "Run X", "Ask Y", "Ship Z".
3. The adversarial review in pm-strategy must require a named, falsifiable assumption (a specific percentage / event / timeframe) — not a category-level risk.
4. After running the quality pass on 3 real products, document specific before/after examples for each changed prompt in `test/quality-examples/` as regression anchors.
5. The pm-prd skill must include a "What will be different in commits after this ships?" field in the Success Criteria section.

### Non-functional requirements
- Changes must remain backward-compatible with existing `.nanopm/` artifact files — new prompts should not invalidate prior artifacts.

---

## Design notes

- The hardest UX decision: how specific is "specific enough" for an artifact section? The quality bar needs to be defined concretely (e.g., "a section passes if it cannot be copy-pasted unchanged into a different product's artifact") — otherwise the quality pass produces a new flavor of generic.
- Adversarial challenges should name a number and a timeframe, not just a category. "Fewer than 3 of 10 users return within 14 days" is useful. "Users may not retain" is not.
- The test suite (`test/`) should grow: tier-1 tests currently validate syntax, not output quality. Consider a tier-3 test that runs the pipeline on a fixture product and asserts that known-weak phrases (e.g., "users may face challenges") do not appear in outputs.
