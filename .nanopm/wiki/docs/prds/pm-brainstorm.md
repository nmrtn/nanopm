## Problem Statement

When a founder wants to think out loud about *what to build next* — a vague feature idea, a
user problem, a "is this even worth it" gut check — they open ChatGPT. That thread knows
nothing about their company, objectives, or code; they re-explain context every time, and a
week later they can't find the thread again (PERSONAS.md: "his last 'strategy doc' is a
4,000-word ChatGPT thread he can't find again"). The cost is twofold: the jam never compounds
(each session starts from zero), and the thinking happens disconnected from the artifacts
nanopm already holds. They want "a thinking partner that pushes back" (PERSONAS.md emotional
JTBD) — not another gated pipeline step, just someone on the product team to riff with who
already knows the situation.

nanopm has a Define/Discover/Plan/Build pipeline but **no informal surface**, on either the
CLI (where Theo lives) or the viewer (where Dani lives). Every existing skill produces a
committed artifact behind a gate. There is nowhere to *just think* with full context loaded.

---

## User Stories

1. As Theo (terminal-native), I want to start a casual chat with a virtual CPO that already
   knows my company and objectives, and resume past jams from my host's session picker, so my
   product thinking accumulates instead of evaporating like an unfindable ChatGPT thread.
2. As Dani (viewer user, non-terminal), I want to open a Brainstorm section that drops me
   straight into a new chat, names and saves the conversation for me automatically, and lists
   my past conversations so I can pick one back up — without ever thinking about session files
   or commands.

founders (KR1, OBJECTIVES.md) voluntarily start a *second* brainstorm session — a resume or a

## Scope

### In scope (v1) — CLI surface (skill `/pm-brainstorm`)
- A daily-ops slash-command skill (outside the gated pipeline, like `/pm-standup`) — the
  in-grain shape; **not** a PATH-installed binary or a `read`-menu shell launcher (neither
  exists in nanopm today; both would be net-new and non-portable).
- Loads company + objectives context via `nanopm_preamble` / CONTEXT-SUMMARY.md — no new ingestion.
- A single jam role: skeptical-but-supportive virtual CPO/PM, voiced per ETHOS.
- **Records its own `brainstorm` state entry** (session id + host `ai-title` + one-line
  summary) so brainstorms are listable across all hosts without parsing host JSONL.
- **Resume leans on the host's native picker** (`claude --resume` / `vibe --resume` /
  `codex resume`) — which already lists sessions by title with keyboard select and does true
  full-transcript resume. The skill may surface a brainstorm-scoped list of its own records via
  `AskUserQuestion` (portable, headless-safe) and point the user to the host resume command.
- Multi-host portable (Claude / Vibe / Codex) from one skill source, per `setup`.

### In scope (v1) — Viewer surface (SwiftUI Brainstorm page; Claude Code backend)
- A **Brainstorm** section that lands **by default on a new, empty chat box** ready for input.
- **Auto-naming, transparent**: read Claude Code's `ai-title` for the conversation — no manual
  naming, no net-new title generation.
- **Conversation list**: past brainstorm conversations, discovered by scanning
  `~/.claude/projects/<encoded-cwd>/*.jsonl` for `ai-title` + last-activity timestamp;
  selecting one **resumes it with full context** via the existing `--resume <id>` path.
- A **free-text composer** (net-new — today the UI only sends structured `nanopm-question`
  answers) and a "keep talking after the model stops" path.
- A **brainstorm persona preamble distinct from the skill question-contract** (the existing
  `interfaceContract` tells the model to ask via JSON blocks and STOP — that would break a
  free chat).
- A **tighter tool allow-list** for brainstorm runs (a conversation persona shouldn't carry the
  full Bash/Write/Edit blast radius).

### Out of scope (v1)
- **Hand-rolled transcript persistence / reload, auto-naming, or a resume engine** — host-native.
- **A PATH-installed `nanopm` launcher / terminal `read` menu** — against the grain; use the
  slash-command + host picker + `AskUserQuestion` instead.
- **Adversarial / falsification gate at session end.**
- **Auto-handoff to pm-prd / pm-discovery** — v1 only *suggests* a next skill in prose.
- **Multi-persona jam partners** (designer / eng / growth).
- **Manual rename / delete / search in the viewer list**, and **multi-host viewer support**
  (the viewer is Claude-Code-only; no host abstraction exists).

## Requirements

### Functional requirements — CLI
1. `/pm-brainstorm` sources `lib/nanopm.sh`, runs `nanopm_preamble`, loads CONTEXT-SUMMARY.md
   + OBJECTIVES.md so the CPO has company + objectives context.
2. On launch, the skill offers (via `AskUserQuestion`, reading its own `brainstorm` state
   records) to start a new jam or resume a listed one; for a resume it surfaces the host resume
   command (`claude --resume <id>`, etc.). It does NOT reload transcripts itself — the host's
   native resume does (an in-session skill cannot re-drive the host to another session).
3. Free-form, multi-turn conversation in the voice of a CPO that pushes back (problem-first,
   names assumptions) rather than agreeing.
4. The skill writes a `brainstorm` state record (session id + `ai-title`/topic + one-line
   summary) at start and updates the summary at end; it suggests — not forces — a next skill.
5. Installs across all hosts via `setup`; passes `test/skill-syntax.sh`.

### Functional requirements — state layer
6. Add a `brainstorm` type to **both** `bin/nanopm-state-log` (SCHEMAS) and
   `bin/nanopm-state-read` (VALID_TYPES) with fields `topic`, `summary`, `session`. The
   validator rejects unknown types/fields, so this is required, and a `setup` re-run ships it.
7. `lib/nanopm.sh` gains `nanopm_brainstorm_record` / `nanopm_brainstorm_list` wrappers over
   the state binaries (mirroring the existing `nanopm_state_log` pattern).

### Functional requirements — Viewer (Claude Code backend)
8. A Brainstorm section defaulting to a new empty conversation on open.
9. A cross-launch session-discovery layer: scan `~/.claude/projects/<encoded-cwd>/*.jsonl`,
   read `ai-title` (fallback: first user message) + last timestamp, render the list; selecting
   one resumes via `--resume <id>`. (The encoded dir = cwd with separators → `-`.)
10. A free-text composer wired to a new `sendMessage(runID, text)` that re-enters
    `startTurn(…, resumeSession:)`; the engine already supports the resume turn.
11. A brainstorm persona preamble replacing the question-contract for these runs, and a tighter
    tool allow-list than the default `Read Edit Write Glob Grep Bash WebFetch WebSearch …`.
12. On resuming a persisted session, either replay the JSONL into transcript bubbles or show a
    "continuing an earlier conversation" header with only new turns (UX call — see below).

### Host resume support (verified 2026-06-16)
| Host | Resume | Auto-title | Viewer support |
|------|--------|-----------|----------------|
| Claude Code | `claude --resume [name]` / `/resume` | ✅ `ai-title` in JSONL | ✅ (the only backend) |

| OpenAI Codex | `codex resume [--last\|ID]` / `codex exec resume` | unverified | ❌ |

### Non-functional requirements
- No new connectors / ingestion sources (OBJECTIVES.md anti-goal #4).
- `brainstorm` records live in `~/.nanopm/projects/{slug}/brainstorm.jsonl` (existing state path).
- Viewer remains a throwaway prototype — reuse `RunSessionView` / `ActivityMonitorView`
  patterns; don't gold-plate. Brainstorm runs get a reduced tool allow-list (security).

## The One UX Decision

**CLI resume: native host picker, or skill-rendered brainstorm list?**

Auto-naming is settled (read host `ai-title`); the live decision is how the CLI user resumes.

- **Option A — Lean on the native host picker.** `claude --resume` (and the vibe/codex
  equivalents) already lists sessions by title, keyboard-selectable, with true full-transcript
  resume. Zero build, true resume — but the list is *all* sessions, not brainstorm-scoped.
- **Option B — Skill-rendered brainstorm list via `AskUserQuestion`.** `/pm-brainstorm` reads
  its own `brainstorm` records and shows a brainstorm-only list; the pick maps to a host resume
  command. Scoped and portable/headless-safe, but it can only *point to* the host resume (an
  in-session skill can't reload another session itself), so it's a finder, not the resumer.

**Recommendation: B for discovery, A for the actual resume.** Show the brainstorm-scoped list
(B) so the user finds the right jam, then hand off to the host's native resume (A) for the real
full-context reload. Best of both, and it's the only combination that's portable *and* truly
resumes. (Viewer sub-decision, FR12: replay JSONL into bubbles vs. show-only-new-turns —
recommend show-only-new-turns for v1; replay is polish.)
