# PRD: Location Assumption Test
Generated by /pm-prd on 2026-03-28

## Problem Statement

nanopm's entire distribution strategy relies on one untested assumption: that solo founders who use Claude Code are willing to do their product planning inside the terminal, alongside their code. If planning location is a habit — not an incidental choice — then no amount of artifact quality improvement will fix the adoption problem. Today there is zero external data on where Claude Code users actually do their planning thinking. The cost of not testing this now is committing weeks of artifact quality work to a framing that may be fundamentally misaligned with how the target user operates.

---

## User Stories

- As the nanopm author, I need to know whether Claude Code users do product planning in the terminal so that I can decide whether to invest in artifact quality (confirms embedding) or reframe the product as "generate in terminal, consume anywhere" (contradicts embedding) before spending Q2 capacity.
- As a Claude Code user responding to the poll, I want a simple, honest question I can answer in 5 seconds so that I can give useful signal without feeling like I'm being sold to.

## Success Criteria

| Criteria | How Measured | Target |
|----------|-------------|--------|
| Response volume | Count of poll responses within 24 hours | ≥30 responses (minimum for signal; <30 = not enough data, re-run or extend window) |
| Location distribution | % of respondents selecting CLI/terminal | ≥30% → embedding assumption holds; <30% → framing pivot required before distribution |
| Decision reached | Strategy.md updated with poll result and next action | Within 48 hours of poll closing |
| What will be different in commits after this ships? | Review git log 7 days post-poll | STRATEGY.md will contain one of two things: (a) "Poll confirmed ≥30% CLI/terminal — proceed with current embedding strategy" OR (b) "Poll failed threshold — framing updated to generate-in-terminal-consume-anywhere; STRATEGY.md How We Win #3 revised." Either outcome is a commit. No commit = poll wasn't acted on. |

**Anti-goals:** Getting a large sample size for statistical significance is NOT a goal for v1. This is a directional signal, not a market research study. 30+ responses is enough to move.

## Scope

### In scope (v1)
- One post in one Claude Code community (Discord or GitHub Discussions — whichever has higher active engagement at time of posting)
- One question: "Where do you do your product planning? Terminal/CLI | Notion/docs | ChatGPT | Somewhere else"
- 24-hour window
- Record raw response distribution
- Update STRATEGY.md with result and next action within 48 hours

### Out of scope (v1)
- Follow-up interviews or DMs to respondents — revisit if poll result is ambiguous or borderline (25–35% CLI)
- Posting to multiple communities (indie hackers, Twitter/X, Hacker News) — revisit in distribution pass after strategy is confirmed
- Building instrumentation or tracking infrastructure — manual count is sufficient for this signal
- Segmenting responses by user type (solo founder vs team PM vs hobbyist) — too much friction on a 5-second poll

---

## Requirements

### Functional requirements
1. Poll question must be unambiguous and answerable without product context — a stranger who has never heard of nanopm should be able to answer it
2. Response options must be mutually exclusive — if someone plans in both terminal AND Notion, they pick their primary
3. Post must include minimal framing context to avoid priming: no "I'm building a terminal PM tool" language before the question
4. Results must be captured before the 24-hour window closes — set a calendar reminder

### Non-functional requirements
- Zero code changes required — this is a research task, not a build task
- Total time to complete: under 2 hours (draft post, publish, wait, record results, update STRATEGY.md)
