# PRD: Local Telemetry Logging
Generated by /pm-prd on 2026-04-07
Project: nanopm
Feature: Local skill usage logging (`nanopm-telemetry-log` + `nanopm-analytics`)
Ties to: Objective 2 KR1 — README matches code

---

## Problem

The README documents three telemetry tiers, an analytics CLI (`~/.nanopm/bin/nanopm-analytics`), and `~/.nanopm/bin/nanopm-telemetry-log`. None of these exist. If a user installs nanopm and follows the README instructions to view their usage stats, they'll find nothing. gstack's telemetry is opt-in but actually works. nanopm's is documented but not built. This is a trust gap that compounds as users arrive.

The SKILL.md files now have telemetry blocks that call `~/.nanopm/bin/nanopm-telemetry-log` — so the hooks are being added. The binary needs to exist and work for the docs to stop lying.

---

## User stories

1. As a solo builder using nanopm for a month, I want to run `~/.nanopm/bin/nanopm-analytics` and see which skills I've used and how long they took — so I can understand my own planning habits.

2. As a new user reading the README, I want the telemetry section to describe something that actually works — so I can configure it and trust that the tool is honest about what it does.

3. As the builder of nanopm, I want local usage logs to exist — so I can eventually understand aggregate usage patterns when users opt in to sharing them.

## What we're building

### `~/.nanopm/bin/nanopm-telemetry-log` (the writer)

A bash script called by each skill's telemetry block:

```bash
nanopm-telemetry-log \
  --skill "pm-audit" \
  --duration 142 \
  --outcome "success" \
  --session-id "abc123"
```

Appends one JSONL line to `~/.nanopm/analytics/skill-usage.jsonl`:

```json
{"ts":"2026-04-07T20:00:00Z","skill":"pm-audit","duration_s":142,"outcome":"success","session_id":"abc123","version":"0.4.0","os":"darwin","arch":"arm64"}
```

Rules:
- No network calls. Local only. Always.
- Silent fail if disk full or permissions issue (`|| true` already in skill blocks)
- Creates `~/.nanopm/analytics/` if it doesn't exist
- Appends only — never reads, never modifies existing entries

### `~/.nanopm/bin/nanopm-analytics` (the reader)

A bash script that reads `skill-usage.jsonl` and prints a summary:

```
~/.nanopm/bin/nanopm-analytics        # last 7 days (default)
~/.nanopm/bin/nanopm-analytics 30d    # last 30 days
~/.nanopm/bin/nanopm-analytics all    # all time
```

Output format:
```
nanopm usage — last 7 days

  pm-audit          3 runs    avg 2m 18s    last: 2026-04-07
  pm-strategy       2 runs    avg 4m 12s    last: 2026-04-07
  pm-competitors-intel  2 runs    avg 1m 44s    last: 2026-04-07
  pm-roadmap        1 run     avg 0m 52s    last: 2026-04-06
  pm-scan           1 run     avg 1m 03s    last: 2026-04-01

  Total: 9 runs · Most used: pm-audit · Total time: 18m 22s
```

Rules:
- Pure bash + python3 (already a dependency)
- If `skill-usage.jsonl` doesn't exist: print "No usage data yet. Run any /pm-* skill to start logging."
- Handle empty/corrupt lines gracefully

### Telemetry config (honour existing README docs)

The three tiers documented in README:
- `off` — skip writing to JSONL entirely (check `nanopm_config_get telemetry` in the log script)
- `anonymous` — write to local JSONL (default)
- `community` — reserved for future network upload; for now, treat as `anonymous`

If telemetry is `off`, `nanopm-telemetry-log` exits 0 immediately without writing anything.

### Install

`setup` script copies both binaries to `~/.nanopm/bin/` and makes them executable. `~/.nanopm/analytics/` is created on first log write.

## What we're NOT building

- **No network calls in this version** — community tier writes local only; network upload is a future feature once there are enough users to make aggregate stats meaningful
- **No dashboard or web UI** — the CLI `nanopm-analytics` is the full viewer this quarter
- **No log rotation or compaction** — JSONL grows indefinitely; acceptable for now at the usage volumes expected in Q2

## The one UX decision

Local-only by default, with a config key to opt out (`telemetry=off`). Not opt-in — opt-out. Rationale: the data is local, nobody else sees it, and it makes `nanopm-analytics` useful from day one without asking the user to configure anything. If a user doesn't want even local logging, one config line disables it. This matches the pattern gstack uses for their telemetry.
