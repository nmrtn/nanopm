# PRD: Memory as a compounding LLM-wiki
Generated by /pm-prd on 2026-06-23
Project: nanopm
Status: DRAFT

> Companion design doc (shareable, narrative): `docs/memory-wiki-redesign.md`.
> This PRD is the buildable spec; the design doc is the "why" for Nico.

---

## Problem Statement

Every nanopm skill run loads the **entire raw event log** (`nanopm_context_all`, the
`~/.nanopm/memory/{slug}.jsonl` file) on top of the two consolidated summaries. That log is
append-only and unbounded, never deduplicated, partially corrupted (values with newlines split a
record across lines, so invalid JSON is fed into runs), and still carries retired skills
(`pm-scan`, `pm-audit`). Today it's ~50 lines / ~13KB of duplicated, half-broken records injected
into 16 skills on every invocation. It is not memory — it is a transcript we re-read in full each
time, and it grows forever.

This hits the **primary persona directly**: Terminal-native Theo's job-to-be-done explicitly
includes *"make the planning compound across sessions instead of evaporating with the ChatGPT
thread"* (PERSONAS.md). The current mechanism does the opposite of compounding — it accumulates.
Theo's workaround cost is *"months of compounding wrong-direction output, and the time tax of
bouncing four tools every planning cycle"* (PERSONAS.md). A memory that bloats and drifts is the
failure mode the product exists to kill, showing up inside the product itself.

The ambition behind the always-loaded context is right (keep Nano on a shared baseline so
downstream work doesn't drift). The **mechanism** is naive: "keep everything + reload everything"
is not memory.

---

## User Stories

1. As a maintainer running the full pipeline (Theo / Guillaume / Nico), I want each skill to start
   from a bounded, consolidated, de-duplicated context so that prior decisions carry forward
   correctly without re-reading a growing raw log.
2. As an ingesting skill (Discover phase), I want a new source to be **integrated across the
   relevant entity pages** rather than dumped as one more log line, so that knowledge compounds
   instead of piling up.
3. As a maintainer, I want to ask "what do we know about persona X / competitor Y / opportunity Z?"
   and get one maintained page with provenance, so that signal isn't scattered across
   FEEDBACK/DATA/COMPETITORS docs.

## Success Criteria

| Criteria | How Measured | Target |
|----------|-------------|--------|
| Context loaded at skill start is bounded | Token/byte count of preamble-loaded context (index + overviews) across 10 consecutive runs | Bounded and roughly flat run-over-run; no growth with history depth (vs today's monotonic growth) |
| Raw log no longer auto-loaded | `nanopm_context_all` removed from every skill preamble path | 0 skills load the raw log at startup |
| No duplicate/corrupted records reach a run | Inspect loaded context for split-JSON lines and repeated near-identical entries | 0 corrupted lines; duplicates collapsed by provenance-keyed dedup |
| Knowledge compounds in entity pages | After a Discover run, the touched entity page(s) reflect the new source with a provenance marker | ≥1 entity page updated (not a new orphan) per ingest, with source citation |
| Ingest is correct, not just present (mirrors Falsification b) | `git diff` on `.nanopm/wiki/` after each ingest: no duplicate provenance keys within a page, no malformed frontmatter, no merge-conflict markers | 0 duplicate keys / 0 corrupt pages per 10 ingests |
| Wiki stays healthy | Lint/sleep agent reports contradictions / stale / orphans / gaps | Lint runs and produces a report; 0 unresolved orphan pages after a lint pass |
| What will be different in commits after this ships? | Review git log 7 days post-ship | `lib/nanopm.sh` loses `nanopm_context_all` from the load path and gains a generic ingest/bookkeeper dispatch + `nanopm_load_index`; a new `NANOPM-WIKI.md` schema lands; skill `SKILL.md` Phase 0 blocks drop `nanopm_context_all` calls; `.nanopm/wiki/` (index.md, log.md, overview/, entities/) appears; viewer `Models.swift` path mappings updated for the wiki layout; `test/` gains a wiki-integrity test |

**Anti-goals (NOT success for v1):**
- A bigger context window loaded at start (the point is *less*, consolidated — not *more*).
- A vector DB or any server/daemon dependency (breaks the file-based, host-agnostic DNA).
- Perfect cross-host parity on day one — graceful degradation is acceptable (see Dependencies).

## Falsification

**Within 21 days of shipping the wiki redesign to `main`**, the central bet — *a maintained,
consolidated wiki beats the always-loaded raw log for keeping agents on-context, and the
ingest/bookkeeper compounds knowledge without corrupting it* — is **wrong** if, with the **two
maintainers (Guillaume + Nico) running the full pipeline on the nanopm repo**, either: (a) **≥2 of
10 consecutive skill runs** surface stale or contradictory baseline context that the old
always-loaded log would have carried correctly (observed in run output / the resulting artifact),
or (b) the ingest/bookkeeper subagent **corrupts or duplicates ≥1 entity page per 10 ingests**
(observed in `git diff` on `.nanopm/wiki/`). Either signal means the consolidation membrane is
losing or mangling knowledge the naive log preserved, and we revert the load-path change.

*(solo-fast mode: "observable behavior" = maintainer review of run output + git diff, small-N is
valid.)*

## Scope

### In scope (v1)

**1. The three-layer wiki under `.nanopm/`**
- `raw/` — immutable sources (connector pulls, interviews, and the existing typed JSONL event log
  moved here). Read, never rewritten. Never auto-loaded.
- `wiki/` — LLM-owned markdown: `index.md` (catalog, always loaded), `log.md` (chronological,
  greppable `## [date] op | Title`), `overview/company.md` (= today's CONTEXT-SUMMARY) +
  `overview/current-work.md` (= today's PLAN-SUMMARY), `entities/` (personas, competitors,
  opportunities, objectives, features, people), `docs/` (skill outputs filed back as views).
- `NANOPM-WIKI.md` — the schema/librarian: conventions, page formats, frontmatter spec, and the
  ingest/query/lint workflows with subagent dispatch + fallback rules.

**2. Loading rule change (the core cut)**
- Remove `nanopm_context_all` from the preamble load path.
- `nanopm_preamble` loads `wiki/index.md` + the two overviews only. Everything else is read
  on-demand by the page named in the index.
- The typed JSONL log becomes a raw episodic source, queried only by skills that need it
  (pm-retro, lint).

**3. Entity pages across all sections** with provenance-keyed claims:
- Each claim carries a source citation in frontmatter → ingest is idempotent/commutative; grep the
  citation before writing = dedup at write time (keyed on source identity, not text proximity).
- Supersede, don't delete — superseded claims are marked and kept with history, not erased.

**4. Generic ingest/bookkeeper subagent** (generalizes the existing summary-regen subagent + the
opportunities dedup agent), parameterized by section:
- Reads raw in its own context, integrates across entity pages, updates `index.md` + `log.md`,
  refreshes the section overview. Raw never touches the main run.
- **Confidence-gated writes:** high-confidence updates auto-apply; ambiguous ones (e.g. a strategy
  reversal) go to a review surface for the human. No silent overwrite of correct knowledge.
- **Single-writer-per-file:** each ingest owns its target page; ROADMAP/overview edits are
  serialized through one writer.

**5. Lint / sleep agent** — a maintenance pass (promotes today's `nanopm_staleness_check` from a
warning into action): contradictions, stale claims, orphan pages, missing-but-referenced concepts,
gaps. Runs when staleness fires or on demand; emits a report, opens fixes as gated writes.

**6. Typed relationship edges** (kept in v1 per scope decision): pages declare typed outbound
relationships in frontmatter (`relates_to: [{page, rel}]` with `rel` ∈
contradicts/supports/extends/supersedes/responds-to). The schema (`NANOPM-WIKI.md`) owns the
permitted vocabulary and the rule that a *missing expected contradiction* is the lint smell, not a
present one. Navigable via the index; full graph query deferred (see Out of scope).

**7. Migration of existing docs into the wiki layout** — CONTEXT-SUMMARY → `overview/company.md`,
PLAN-SUMMARY → `overview/current-work.md`, opportunities/ stays (it is already the conformant wiki
section and the template), Define/Plan docs → `wiki/docs/`. Zero content loss. **One-time cleanup
of the existing raw JSONL log** — collapse duplicates and repair the split-JSON records before it
moves to `raw/`, so the migration fixes today's corruption rather than carrying it forward.

**8. Viewer rework** (kept in v1 per scope decision): update the SwiftUI viewer to read the new
`wiki/` layout natively — `Models.swift` path conventions (`ReasoningFiles`, `PhaseMapper`, the
CONTEXT/PLAN → overview mappings), the Plan overview render, and the opportunities surface. Done
properly, not via a throwaway shim. **Sequenced (see Decisions):** native overviews + entity pages
first, opportunities surface untouched, then the rest of the rework — all within v1.

### Out of scope (v1) — each gets a minimal PRD + a ROADMAP LATER entry (separate subagents)
- **Search engine over the wiki** (qmd / BM25 / vector + reranking) — revisit when `index.md`
  stops scaling (well past current tens-of-pages). Markdown + index is enough now.
- **Multi-writer git machinery** (worktree-per-agent isolation + deterministic merge drivers) —
  revisit when concurrent ingest contention becomes real; single-writer-per-file partitioning
  covers v1.

## Requirements

### Functional requirements
1. `nanopm_preamble` MUST load `wiki/index.md` + `wiki/overview/company.md` +
   `wiki/overview/current-work.md`, and MUST NOT call `nanopm_context_all`.
2. A new loader (`nanopm_load_index` or equivalent) MUST emit the index catalog; existing
   `nanopm_load_context` / `nanopm_load_plan` MUST point at the new overview paths.
3. `nanopm_context_all` MUST be removed from all skill Phase-0 blocks (21 skills). Skills needing
   episodic detail (pm-retro) MUST query the raw log explicitly.
4. The ingest/bookkeeper subagent MUST: dedup by source citation before writing, supersede rather
   than delete, update `index.md` + `log.md`, and refresh the touched section overview.
5. Writes MUST be confidence-gated: ambiguous writes routed to a human-review surface, not
   auto-applied.
6. `NANOPM-WIKI.md` MUST define page formats, the frontmatter/provenance spec, the typed-edge
   vocabulary, and the ingest/query/lint workflows including subagent dispatch + host fallback. It
   is **deliverable #1** — everything else keys off it. Note: the gated subagent dispatch + host
   fallback is a **net-new pattern** (no current skill conditionally dispatches Agent — today's
   skills assume it is present), so it needs its own small cross-host (Claude/Vibe/Codex) test.
7. The lint/sleep agent MUST detect contradictions, stale claims, orphans, and gaps, and emit a
   report.
8. Migration MUST preserve all existing content and keep the opportunities wiki working unchanged.
9. The viewer MUST render the new `wiki/` layout (overviews, entity pages, opportunities) without
   regression to existing surfaces.

### Non-functional requirements
- **Host-agnostic:** works on Claude / Vibe / Codex. Subagent dispatch is **gated** (Agent tool
  present + section touched) and **degrades gracefully** (no Agent tool → main agent does a
  lightweight inline update or marks the overview stale; control always stays with the main agent).
- **No new infrastructure:** markdown + git only. Any index/search artifact must remain disposable
  and rebuildable; markdown+git is the single source of truth.
- **Deterministic startup:** loading is bash, never a subagent on the critical path.

## The One UX Decision

The "UX" here is the maintainer + agent experience. The one decision that, made wrong, invalidates
the rollout:

**Option A — Hard cut.** Remove `nanopm_context_all` from the load path in one change; all 21
skills migrate to index+overviews at once. Clean, no dual-mode, but a single big breaking commit:
any skill that still leaned on raw-log specifics breaks until fixed.

**Option B — Deprecate with a shim.** `nanopm_context_all` is repointed during a transition to
return only the index + latest-entry-per-skill (a bounded view), skills migrate incrementally, then
the shim is deleted. Safer rollout, but two code paths live at once and the "naive load" lingers
longer.

Tradeoff: A optimizes for a clean end-state and speed (fits solo-fast, one repo, two maintainers);
B optimizes for not breaking a mid-pipeline run. **Decision: Option A — hard cut (accepted
2026-06-23).** There is exactly one consuming repo and two maintainers who can fix forward, and the
whole point is to stop loading the raw log — a shim that keeps loading a bounded slice of it defeats
the cut. All 21 skills migrate to index+overviews in the load-path wave.

## Decisions (resolved 2026-06-23)

| Question | Decision | Rationale |
|----------|----------|-----------|
| Hard cut vs shim for retiring `nanopm_context_all` | **Hard cut (Option A)** — remove it from the load path in one change; all 21 skills migrate to index+overviews together | One consuming repo, two maintainers who can fix forward; a shim that keeps loading a bounded slice of the log defeats the cut |
| Keep the typed JSONL log as the canonical raw episodic source? | **Yes** — the typed JSONL stays (schema-validated provenance) and moves to `raw/`; `log.md` is the human/LLM-facing chronological view generated alongside it, not a replacement | Don't discard machine-validated records; pm-retro and lint read the JSONL, humans skim `log.md` |
| Typed-edge vocabulary governance (the gist's named caveat) | **Fixed small vocabulary in `NANOPM-WIKI.md`** (contradicts / supports / extends / supersedes / responds-to); the lint/sleep agent flags any out-of-vocabulary `rel` and any ambiguous extends-vs-supersedes for human arbitration during lint review | Keeps edge typing maintainable on every ingest; turns drift into a lint check rather than open-ended ontology work |
| Viewer: full rework now vs phased | **Phased within v1** — render the new overviews + entity pages natively first (opportunities surface untouched, it already conforms), then complete the rest of the viewer rework. Sequencing, not deferral — it stays in v1 | Matches the wave order; the viewer is a throwaway prototype, so don't block the memory cut on its full rework |

## Dependencies

- **`lib/nanopm.sh`** — `nanopm_preamble`, `nanopm_load_context`, `nanopm_load_plan`,
  `nanopm_context_all`, `nanopm_context_append`, `nanopm_staleness_check`. Core of the change.
- **All 21 skill `SKILL.md` files** — Phase-0 blocks that call `nanopm_context_all` and set
  `_CHALLENGES`/other path vars must migrate to index+overviews + on-demand reads.
- **The viewer** (`viewer/.../Models.swift`) — **load-bearing "change one, change both" couplings**
  (per CLAUDE.md): `ReasoningFiles` mirrors `nanopm_reasoning_path` (lib), and `PhaseMapper` mirrors
  the `PLAN-SUMMARY.md → .plan` convention. Every path the redesign moves (overviews, reasoning
  sidecars, the new entity/`docs` dirs) MUST be updated on both sides in lockstep or the viewer
  silently mis-renders.
- **Typed state validator** (`bin/nanopm-state-log`) — provenance/confidence frontmatter should
  stay consistent with the typed records.
- **The opportunities wiki** (`.nanopm/opportunities/`) — the reference implementation and template
  for entity pages; must keep working through migration.
- **`setup`** — installs `lib/` and writes the install-source provenance marker; must lay down the
  new `NANOPM-WIKI.md` and wiki scaffolding.
- **Tests** — `test/context-threading.e2e.sh` (must be updated for the new load path) +
  a new wiki-integrity check.
