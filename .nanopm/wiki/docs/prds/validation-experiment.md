## Problem Statement

Solo founders running AI coding agents (Claude Code, Vibe, Codex) currently keep product context in ChatGPT chats, scattered Notion docs, or nowhere at all — each new session starts cold and re-explains the same product to the same model. The workaround costs them ~10–20 minutes per session re-establishing context, and worse, it costs them decision continuity: yesterday's bet evaporates because nothing typed survives the session. nanopm shipped typed-state memory in v0.6.0 to solve exactly this — but zero external users have verified that the memory pull is real. The product is technically coherent and has been used by one person: the builder. Every artifact downstream of this validation gap is built on an untested assumption.

> "Have any of your close-friend users run the full pipeline unprompted — not as a demo, not to help you test — in the last 30 days?" — from FEEDBACK.md top unaddressed signal (1 finding, High severity, ❌ not addressed in roadmap)

This PRD is a prior attempt's redo. `behavioral-demand-test.md` was the April PRD for the same validation — it was never shipped. We have spent 8 weeks adding infrastructure (typed state, gates, multi-host, telemetry purge, portability hotfixes) instead of running the test.

## User Stories

- As a **solo founder running `/pm-run` for the first time on my own product**, I want to see real value from the pipeline — concrete artifacts that change what I'd build next — so that I have a reason to come back tomorrow.
- As a **first-time completer returning a week later**, I want the next skill I run to know what I decided last time — without me re-explaining — so that the typed memory pays off observably.

## Success Criteria

| Criteria | How Measured | Target |
|---|---|---|
| First-time pipeline completions by external users | Count of unique non-`nanopm` slugs that produce a full pm-run trail in `~/.nanopm/projects/{slug}/timeline.jsonl` (skill=`pm-run`, event=`completed`) | **≥10 within 14 days of poll launch** |
| Second-skill invocations that read prior decision state | Count of unique non-`nanopm` slugs whose 2nd skill invocation triggers a `memory-read` event in `timeline.jsonl` (new event type wired as Sub-task 1 of the roadmap NOW item) | **≥3 of the first 10 completers, within 14 days of their own first run** |
| Self-reported "this changed what I'd build next" | Direct message / reply / GitHub issue from completer naming the artifact or decision that changed their thinking | **≥1 explicit confirmation by day 14** (Obj 1 KR2) |
| What will be different in commits after this ships? | Review git log 14 days post-poll-launch | New `bin/nanopm-memory-read-ping` (or equivalent in `lib/nanopm.sh`); event=`memory-read` lines in test fixtures; a results write-up under `.nanopm/intel/VALIDATION-{date}.md`; one of: marketplace listing branch OR `library-mode-pivot/` branch OR a CHANGELOG entry stating the project is parked. |

**Anti-goals:** *more* installs without second-skill memory-reads do NOT count. 50 first-time completers with 0 returns falsifies the bet just as cleanly as 5 first-time completers with 0 returns.

## Falsification

**Required field — gated.**

Fewer than 3 of the first 10 external solo founders (running on Claude Code, Vibe, or Codex) who complete a `/pm-run` invoke any second nanopm skill that emits a `memory-read` event reading from `~/.nanopm/projects/{slug}/decision.jsonl` within 14 days of their first `pm-run` completion timestamp.

If this happens, the central bet — *"typed-state memory is the pull"* — is wrong. Options on falsification: (a) pivot to library-mode (nanopm exposed as a function library other agents call directly), (b) reframe as a one-shot planning tool (give up on the return-loop bet), or (c) park the project.

## Scope

### In scope (v1)

- A **return-ping mechanism**: small change to `lib/nanopm.sh` (or a new `bin/nanopm-memory-read-ping`) that emits a `timeline.jsonl` event of shape `{"skill":"X","event":"memory-read","branch":"...","project":"..."}` whenever a skill calls `nanopm_state_read` against an existing `decision.jsonl` that has prior entries from a different session.
- **Poll posts** across the 4 channels named in STRATEGY.md (Claude Code Discord, r/ClaudeAI, "Show HN", Vibe community). One canonical post template, copy-paste-ready, links to repo + install command.
- **Manual check-in cadence**: builder DMs or replies to first 10 completers at +24h, +72h, +7d, +14d. **Only to ask "did anything in the output change what you'd build?" — NOT to push them toward a second skill** (see UX Decision below).
- **Results write-up** at `.nanopm/intel/VALIDATION-{date}.md` after 14 days. Pass/fail verdict, raw counts, qualitative quotes.
- **Active install support**: respond to GitHub issues and DMs about setup failures within 24h. Install friction is not a test of the bet — fix it.
- **Tracking after day 14**: keep collecting data on the same cohort past the 14-day threshold. The 14-day window is the *success threshold*, not the data cutoff. Tail signal (week 3, week 4) is qualitative bonus.

### Out of scope (v1)

- **Paid promotion** (HN front-page bump, Reddit ads, sponsored Discord posts) — organic only. Revisit only if organic posts get <10 impressions each.
- **Analytics dashboard / UI** — read raw `timeline.jsonl` + `decision.jsonl`. Use a manual spreadsheet for the 10 completers. Revisit only if we run a second validation cohort.
- **Reformulating the README** — locked by STRATEGY.md `not-polished-marketing` scope-out.
- **Adding new skills / connectors based on poll feedback** — locked by STRATEGY.md `not-49-skill-catalog` and `not-new-connectors` scope-outs.

## Requirements

### Functional requirements

1. `lib/nanopm.sh` (or a new helper binary) exposes a `nanopm_emit_memory_read` function that detects when `nanopm_state_read --type decision` was called against a `decision.jsonl` with ≥1 entry from a session older than 30 minutes (i.e., not the current session's writes), and emits `event=memory-read` to `timeline.jsonl` for that project.
2. Skills that call `nanopm_state_read` against `decision.jsonl` invoke `nanopm_emit_memory_read` afterward (or the call is wrapped in `nanopm_state_read` itself — implementation choice).
3. A canonical poll post template is written to `.nanopm/intel/POLL-TEMPLATE-{date}.md`. Same wording across all 4 channels (only channel-specific tone tweaks allowed).
4. Posts are published within 7 days of PRD acceptance (no later).
5. Builder check-in DMs follow a fixed script (committed to the validation doc) — no improvisation about "have you tried running pm-strategy?"
6. Results write-up is published at `.nanopm/intel/VALIDATION-{date}.md` no later than 21 days after poll launch (7-day buffer past the 14-day threshold).

### Non-functional requirements

- Privacy: the memory-read event captures only `skill`, `event`, `branch`, `project_slug_hash` — **not the raw project name**. Hash the slug at emit time. Avoid the same telemetry-leakage problem gstack just shipped tests for.
- Reversibility: if validation fails, the memory-read instrumentation must be trivially removable (single function, single call site per skill).

## The One UX Decision

**Should the builder DM / nudge completers toward a second skill at the 24h, 72h, 7d, or 14d checkpoint?**

- **Option A: Silent observation.** No nudges, no "have you tried /pm-strategy?", no follow-up emails suggesting a second skill. Pure organic pull. — **Risk:** the n=10 cohort might be too small to see the pull signal even if it exists. Could falsify a bet that's actually right.
- **Option B: Soft check-in.** Ask "did anything in pm-run's output change what you'd build?" but do NOT mention specific other skills. The wording avoids steering them toward `decision.jsonl`-reading skills specifically. — **Risk:** any framing nudges behavior; even open-ended check-ins prime the user.

**Tradeoff:** Option A has cleaner signal; Option B has more responses. **Recommendation: Option B.** The cost of missing a real pull-signal due to small-n silence is higher than the cost of mild priming. The check-in script must be committed and not improvised. Specifically: NEVER mention `/pm-strategy`, `/pm-prd`, or any skill name in the check-in DM. Ask about the OUTPUT, not the WORKFLOW.
