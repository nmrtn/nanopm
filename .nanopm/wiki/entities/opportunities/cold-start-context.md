## 1. Problem summary
The bet is that context compounds across sessions — that each PM run builds on a shared, current baseline rather than rebuilding understanding from scratch. If that compounding doesn't actually hold, the builder re-establishes who the company is, who it serves, and what the product does at the start of every iteration. The loop never accelerates; it just restarts.

## 2. Value to the user
### Job to be done
Pick up a new session and immediately work on top of an accurate, current picture of the company and product, without re-explaining the basics each time. The alternative today is re-stating context per session (or per skill), which is repetitive and where divergence between runs creeps in.
### Where we fall short
**No compounding baseline across sessions**
If state doesn't carry a current, consolidated company + product picture into each run, every iteration pays the same cold-start cost instead of getting cheaper over time.

**Divergence between runs**
Without one shared baseline feeding every skill, separate runs can drift apart in what they assume about the company, so outputs don't cohere across the loop.

## 3. Value to the company
Compounding context across sessions is the stated retention lever — the reason the loop gets more valuable the longer it runs. If iterations start cold, there's no compounding, no retention advantage, and the product reduces to a set of one-off generators. This is the difference between a loop and a toolbox.
