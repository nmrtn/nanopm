# PRD: Define — company & product context section
Generated by /pm-prd on 2026-06-11
Project: nanopm
Status: SHIPPED — v0.10.0 · PR https://github.com/nmrtn/nanopm/pull/20 (2026-06-11)

---

## Problem Statement

When a CPO or PM joins a company — or a founder starts a new product — there is a body of
foundational context they must absorb *before* any real PM work is possible: what the company
is and how it makes money, who's who internally, who it's built for, and what the product actually
is and how it works. nanopm has no clean layer for this today. The context exists in scattered,
half-formed pieces: a manual `CONTEXT.md` Q&A at the repo root, `pm-scan` (which reverse-engineers
the product from code but drifts into strategic judgment), and `pm-personas` (which covers only the
*who-we-build-for*). The result is that downstream skills (audit, strategy, roadmap, PRD) build on
memory and self-report instead of a stated, reusable ground truth.

The pain is sharpest in two cases: (1) joining an **existing** codebase, where you need an honest
product map without an audit's opinions mixed in; and (2) starting **greenfield**, where there is
no code to read and the company/product must be *defined* before anything else. nanopm currently
forces an either/or branch (`pm-scan` for existing, `pm-discovery` for greenfield) and conflates
"understand the product" with "discover what to build."

A concrete symptom of the current confusion: `pm-scan` and `pm-audit` overlap on three of their
sections — *what you're building*, *who it's for*, and *the biggest gap* — because `pm-scan` was
built as the audit's feeder ("Pre-filled for pm-audit") but grew judgment of its own. They aren't
badly designed; they're badly *partitioned*.

---

## User Stories

- As a **CPO joining an existing company**, I want nanopm to produce a structured company +
  product context (the business, the org, the personas, an honest product map) and a first critical
  read, so I understand the terrain before I plan.
- As a **founder starting greenfield**, I want to *define* my company and product from scratch
  through a guided interview, so every downstream skill has a stated ground truth instead of
  re-asking me each session.
- As a **PM on an existing product**, I want one product-mapping skill that reads the code *and*
  the public site/positioning, so the map reflects both what's built and how it's sold.

---

## Solution shape

A new **Define** phase, placed first: `Define → Discover → Plan → Build`. It is the orientation
layer — *map the terrain, then form a first judgment* — and runs as an arc:

| # | Skill | Output | Notes |
|---|-------|--------|-------|
| 1 | `/pm-vision-mission` | `VISION-MISSION.md` | Mission, vision, values, company stage |
| 2 | `/pm-business-model` | `BUSINESS-MODEL.md` | Business model, pricing, packaging, GTM motion |
| 3 | `/pm-org` | `ORG.md` | Org map, key roles, decision-makers |
| 4 | `/pm-product` | `PRODUCT.md` | Deep product map. **Absorbs the now-retired `pm-scan`**; web research on site/positioning when the company exists |
| 5 | `/pm-personas` | `PERSONAS.md` | **Moved in from Discover**; reverse-engineers from `PRODUCT.md` |
| 6 | `/pm-audit` | `AUDIT.md` | **Moved in from Discover**; re-partitioned to read `PRODUCT.md` + company docs → purely evaluative (gap + the question you're avoiding) |

Every skill is **dual-mode** (auto-detect existing vs greenfield, reusing the `pm-personas` pattern:
reverse-engineer from code/web when material exists, interview from scratch when empty).

Moving `pm-scan` (retired), `pm-personas`, and `pm-audit` out of Discover leaves Discover as the
three external signals — market (`pm-competitors-intel`), user research (`pm-user-feedback`,
`pm-interview`), and data (`pm-data`) — which is the direction the founder wants Discover to take.
The remaining Discover cleanup (e.g. `pm-discovery`'s role) is deferred.

**Flow — advisory, not a hard gate (decided).** `pm-run` runs Define first by default and the app
leads with it, but nothing blocks a user who jumps straight to Discover/Plan. Downstream skills that
read a Define doc warn-but-proceed when it's missing and offer to run the relevant Define skill
inline. This keeps the falsification measurable — a hard gate would force the metric to ~100% and
destroy the signal about whether Define is actually wanted — and respects the ETHOS "ship then learn"
posture (the metric, not the lock, tells us if Define earns its place).

- Building market/competitive analysis into Define — that's a Discover signal, deliberately excluded.
- Turning the six Define skills into a six-form interview slog (see The One UX Decision).

- Resolving the `CONTEXT.md` deduplication (explicitly deferred).
- The full restructure of what *remains* in Discover (deferred — Define only takes scan/personas/audit out).

## Falsification

Of the next 10 distinct projects where a user starts a nanopm pipeline in the 30 days after the
Define section ships, if fewer than 4 (40%) generate **any** new Define context artifact
(`VISION-MISSION.md`, `BUSINESS-MODEL.md`, `ORG.md`, or `PRODUCT.md`) before reaching a Plan-phase
artifact (`OBJECTIVES.md` / `STRATEGY.md` / `ROADMAP.md`) — measured via local state/telemetry —
then the bet is wrong: establishing company/product context first is not perceived as necessary, and
the section is overhead rather than the foundation the pipeline was missing.

## Scope

### In scope (v1)
- New **Define** phase, first in the order, with the six skills above.
- Four new skills: `/pm-vision-mission`, `/pm-business-model`, `/pm-org`, `/pm-product` — each dual-mode.
- `/pm-product` absorbs `pm-scan`'s code-reading as its existing mode and adds public-site/positioning
  web research; **`pm-scan` is removed**.
- Move `/pm-personas` and `/pm-audit` from Discover into Define.
- Re-partition `pm-audit` to read `PRODUCT.md` + company docs instead of re-deriving facts (purely
  evaluative). Depth this cycle: repoint its fact sections to `PRODUCT.md`; no full rewrite.
- Repoint every former `SCAN.md` reader (`pm-personas`, `pm-audit`, `pm-run`) to `PRODUCT.md`.
- Wire Define into `pm-run` Phase 1 (replaces the scan-vs-discovery branch with "establish context
  first, mode auto-detected per skill") — **advisory**: Define runs by default, the user can skip
  ahead, and missing-context warnings are non-blocking.
- **Backfill existing skills to consume the new Define content (in scope).** `nanopm_context_all` only
  propagates the context-*log* summaries automatically; the rich doc bodies need explicit per-skill
  reads added to each skill's Phase 0 — there is no central switch. Update where it materially improves
  output, every read degrading gracefully if the doc is absent:

  | Skill | Add reads | Why |
  |-------|-----------|-----|
  | `pm-strategy` | `VISION-MISSION`, `BUSINESS-MODEL`, `PRODUCT` | strategy must serve the mission, win commercially, and fit what exists |

  | `pm-prd` | `PRODUCT`, `BUSINESS-MODEL` | scope a feature against what's built + pricing/packaging |
  | `pm-roadmap` | `PRODUCT` | sequence realistically against the current product |
  | `pm-data` | `PRODUCT` | ground which events/features the metrics refer to |
  | `pm-competitors-intel` | `BUSINESS-MODEL`, `PRODUCT` | compare us-vs-them on real positioning |

## Requirements

### Functional requirements
1. Each Define skill auto-detects its mode (reverse-engineer from codebase/web vs interview from
   scratch) and states the chosen mode in one line, reusing the `pm-personas` pattern.
2. `/pm-product` in existing mode reads routes/models/tests/git (the current `pm-scan` logic) **and**
   does web research on the public site (positioning, site structure); in greenfield mode it
   interviews the product concept. Output is a single `PRODUCT.md` regardless of mode. In greenfield
   mode, `PRODUCT.md` is stamped `Completeness: complete` only when all **four essentials** are filled
   with real (non-placeholder) content — **the core problem, the primary user, the product concept, and
   the one core workflow**; otherwise it is stamped `draft`. The stamp is advisory: nothing blocks, but
   downstream skills surface a one-line "planning on a draft product concept" warning when they read a
   `draft` `PRODUCT.md`.
3. `/pm-vision-mission`, `/pm-business-model`, `/pm-org` each work greenfield (interview) and existing
   (web research on about/pricing/team pages, confirmed with the user).
4. Each new skill follows the standard nanopm skill contract: source `lib/nanopm.sh`, run
   `nanopm_preamble`, read prior context (Phase 0), write its `.nanopm/` artifact, save context.
5. `pm-audit` reads `PRODUCT.md` + the company docs if present; its "what you're building / who for"
   sections reference them rather than re-establishing the facts; it keeps its adversarial
   "question you're avoiding" gate.
6. Within Define, ordering is advisory: `pm-product` precedes `pm-personas` (which reads it) and
   `pm-audit` (the capstone, reads everything available). Skills must degrade gracefully if a prior
   Define doc is missing.
7. `pm-run` Phase 1 establishes Define context first by default (mode auto-detected per skill), but is
   **advisory**: the user may skip to Discover/Plan. Any downstream skill that reads a Define doc and
   finds it missing emits a single non-blocking warning, offers to run the relevant Define skill inline,
   and then proceeds with whatever context it has.
8. The viewer renders the Define phase **first** in the sidebar with its six skill rows, maps all
   Define artifacts (`VISION-MISSION.md`, `BUSINESS-MODEL.md`, `ORG.md`, `PRODUCT.md`, `PERSONAS.md`,
   `AUDIT.md`) to `.define`, no longer shows a Codebase Scan row, and still builds cleanly via
   `./build-app.sh`.
9. The app's first-run/guided experience leads with Define (progress badges) but always exposes a
   "Skip for now" affordance — so skipping stays observable in state/telemetry and the falsification
   metric remains meaningful.

### Non-functional requirements
- Multi-host portability preserved (Claude / Vibe / Codex): `AskUserQuestion` headers ≤ 12 chars,
  ≥ 2 options, per portability-v2.
- Backward compatibility: projects with an existing `SCAN.md` still render (mapped under Define);
  the migration must not orphan prior artifacts.

## The One UX Decision

**How heavy is the Define flow — guided auto-derivation, or six manual interviews?**

Define is now six skills. If each is a full from-scratch interview, the foundation phase becomes a
six-form slog that nobody finishes — and the moment that happens, the falsification metric (≥6 of 10
complete a Define artifact before planning) dies and the whole bet fails.

- **Option A — Guided, auto-derive-first.** `pm-run` drives Define as one flow; each skill derives as
  much as it can from code + web and asks the user only to confirm/correct; `pm-audit` runs last as the
  capstone. Lower per-doc fidelity, but it actually gets completed.
- **Option B — Six independent full interviews, à la carte.** Thorough, high-fidelity, but high
  abandonment.

**Recommendation: A.** The phase only pays off if people finish it; completion beats fidelity here.
