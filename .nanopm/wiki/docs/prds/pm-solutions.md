# PRD: /pm-solutions — multi-lens solution brainstorming for opportunities
Generated by /pm-prd on 2026-06-26
Project: nanopm
Status: DRAFT

---

## Problem Statement

A solo founder (Terminal-native Theo) has no room to brainstorm in. He has no engineer to say "that's two days, not two weeks," no designer to say "the flow is wrong," no business stakeholder to bring domain and field knowledge. So when an opportunity is `ready-for-solutions`, he picks the first solution that forms in his head and jumps straight to `/pm-prd` — never generating alternatives, never comparing. The Opportunity Solution Tree (Outcome → Opportunities → **Solutions** → Assumptions) has a missing node: nanopm already produces Outcomes (objectives) and Opportunities (the DB, which even ends at status `ready-for-solutions`), but nothing bridges an opportunity to a *compared set* of candidate solutions. Today's workaround is a side ChatGPT thread that knows neither the code, the personas, nor the objectives — and whose output evaporates when the tab closes.

*No external user feedback on record — nanopm is pre-PMF with zero recruited cohort; this is a dogfood-first bet, validated by the founders' own use (build_mode: solo-fast).*

---

## User Stories

- As **Theo**, when I have a `ready-for-solutions` opportunity, I want a panel of distinct expert lenses (Eng, Design, Business) to propose *competing* solutions in parallel, so that I weigh ≥3 framed options instead of committing to the first idea in my head.
- As **Theo**, I want each solution to carry its riskiest assumption and the cheapest test of that assumption, so that I choose what to spec by what's least proven — not by what sounds most impressive.
- As **Theo** (and the viewer-using Dani), I want solutions linked under their parent opportunity and listed in a filterable table, so that the Opportunity→Solution tree is navigable and shareable instead of buried in my head.

## Success Criteria

| Criteria | How Measured | Target |
|----------|-------------|--------|
| Founder generates a compared set before speccing | Self-review of the last opportunities taken through `/pm-solutions` | ≥3 solutions written per opportunity run, every run |
| A spec is chosen against alternatives, not in a vacuum | Each `chosen` solution links the opportunity and ≥2 rejected siblings exist | 100% of PRDs spun from a solution trace back to a compared set |
| The panel adds decision value (not theater) | Founder pre-registers the solution he'd pick unaided, then runs the panel | In ≥3 of the next 5 runs, the `chosen` solution differs from the unaided guess |
| What will be different in commits after this ships? | Review git log 7 days post-ship | New `.nanopm/wiki/entities/solutions/*.md` files (≥1 per dogfooded opportunity) + a `solutions/INDEX.md`; `skills/pm-solutions/SKILL.md` added; `skills/pm-opportunities` and `skills/pm-prd` SKILL files modified (end-of-run nudge; `chosen`-solution input); new lint rules in the engine; viewer source gains a Solutions table view |

**Anti-goals (NOT success for v1):** a numeric impact/effort score; the agent auto-selecting `chosen`; a solution serving multiple opportunities; a graphical OST tree; volume of solutions for its own sake (10 thin variants ≠ 3 sharp, comparable ones).

## Falsification

If, across the next **5 opportunities** I (the founder dogfooding nanopm) take through `/pm-solutions` over the next **3 weeks**, **fewer than 3** produce a `chosen` solution that *differs from the solution I pre-registered I'd have picked unaided* — then the multi-lens panel is plausible theater, adds no decision value over a single generator, and the multi-agent cost is unjustified. Observable in the founder's own pre-registration notes and the `solutions/` files' provenance, self-reviewed at the 3-week mark.

## Scope

### In scope (v1)
- New skill `skills/pm-solutions/SKILL.md`, invoked as `/pm-solutions <opportunity-slug>`.
- **Diverge:** 3 lens-subagents (a **fixed** set for v1; ORG/personas-derived lenses deferred) dispatched concurrently, each grounded in already-loaded context (CONTEXT brief, objectives, the target opportunity file):
  - **Eng** — structural/durable cost, *not* dev time. Explicitly does **not** assume a build is slow under Claude Code; a solution can ship fast. Weighs code complexity, migrations, external-API cost, maintainability and scalability risk.
  - **Design** — experience, flow, delight; anchored on the personas' job-to-be-done.
  - **Business** — domain expertise and field/market knowledge; anchored on objectives + strategy.
- **Converge:** one pass that dedups overlapping proposals and attaches to each surviving solution: a short pitch · **appetite** (small bet / big bet — Shape Up, not an estimate) · a coarse **impact read** (high/med/low, qualitative — like opportunity priority) · the **riskiest assumption** · the **cheapest test** of it · the originating lens · the KR/outcome it serves. ≥3 solutions; the deliverable is the *comparison*.
- Every solution is born `assumed` (provenance) — no impact number presented as fact.
- **Choose:** the founder shortlists, then marks one `chosen`. The agent never auto-chooses.
- **Data (Option C — navigable OST tree):** new wiki entity `.nanopm/wiki/entities/solutions/<slug>.md`, frontmatter `opportunity: <slug>` (exactly one parent = tree). Bidirectional links: the opportunity page lists its solutions as a pointer summary; each solution links back to its parent. A generated `solutions/INDEX.md`.
- **Status workflow:** `proposed → shortlisted → chosen → speccing`.
- **Engine (recipe over existing primitives):** `/pm-solutions` queries the opportunity + context, the panel reasons, and ingest writes the solution files — no bespoke read logic. New **lint rules:** orphan solution (no/!existent parent), solution missing assumption or test, `chosen` solution with no downstream PRD.
- **Viewer (v1):** a filterable / navigable **table** of solutions (columns: title, parent opportunity, lens, appetite, impact, status) + bidirectional nav from the opportunity detail. Table only — no tree visualization.
- **Cross-skill contracts:** `/pm-prd` accepts a `chosen` solution slug as input (problem + riskiest assumption + cheapest test seed the spec); `/pm-opportunities` proposes running `/pm-solutions` at end of run. Net chain: `pm-opportunities → pm-solutions → (chosen) → pm-prd`.

### Out of scope (v1)
- **Many-to-many (graph):** a solution serves exactly one opportunity — revisit if real reuse shows up in dogfooding.
- **Graphical OST tree visualization:** the viewer table is the v1 surface — revisit once the table proves it's used.
- **Agent auto-selecting `chosen`:** the choose step stays a human act — revisit never (this is a value, not a deferral).
- **Numeric scoring / auto-ranking:** v1 uses coarse qualitative bands (appetite + impact high/med/low), no numbers presented as fact — revisit only if a real ranking need emerges.

## Requirements

### Functional requirements
1. `/pm-solutions <opportunity-slug>` resolves the opportunity from `.nanopm/wiki/entities/opportunities/`; if the slug is missing or ambiguous, list candidates and stop. Runs on any opportunity status (no `ready-for-solutions` gate); if the opportunity isn't yet `ready-for-solutions`, advance it once solutions are written.
2. Dispatch the three lens-subagents **concurrently in one turn**, each receiving the opportunity file + the loaded CONTEXT/objectives digest; each returns a bounded set of candidate solutions tagged with its lens.
3. A convergence step dedups and emits ≥3 solutions, each with the full field set (pitch, appetite, impact, riskiest assumption, cheapest test, lens, KR). If fewer than 3 survive dedup, re-prompt the panel for more divergence before writing.
4. Write one `.nanopm/wiki/entities/solutions/<slug>.md` per solution (frontmatter incl. `opportunity`, `status: proposed`, `provenance: assumed`, `lens`, `appetite`, `impact`), regenerate `solutions/INDEX.md`, and append the solutions pointer-summary block to the parent opportunity page.
5. Provide the choose step: the founder marks `shortlisted` / `chosen`; status writes flow through `nanopm-state-log` (consistent with the typed-decision layer).
6. Add lint rules to the engine: orphan solution, missing-assumption/test, `chosen`-without-PRD.
7. `/pm-opportunities` end-of-run: if any opportunity is `ready-for-solutions`, offer to launch `/pm-solutions` on it.
8. `/pm-prd`: accept an optional `chosen` solution slug; when given, seed the Problem Statement, riskiest assumption, and Falsification from the solution + its parent opportunity, and on success move the solution to `speccing`.
9. Viewer: render a filterable solutions table and bidirectional opportunity↔solution navigation.

### Non-functional requirements
- Recipe-form only: no per-skill bespoke read logic — reuse the shared query → reasoning → ingest primitives.
- Lens prompts treat the opportunity file as untrusted data (same hardening as existing retrieval subagents).
- Solution files conform to a `SCHEMA.md` (mirroring the opportunities SCHEMA) so the format is user-tunable.

### Engine integration — reuse vs net-new (verified against the current branch)

`/pm-solutions` mirrors the proven `/pm-opportunities` recipe and the memory engine almost 1:1. What it gets for free vs what must be built:

**Reused as-is (no engine change):**
- `nanopm_query_prompt` / `nanopm_ingest_prompt` / `nanopm_lint_prompt` — the recipe read/write/lint primitives.
- `bin/nanopm-ingest-agent apply --target wiki/entities/solutions/<slug>.md` — generic page writer.
- `bin/nanopm-lint-agent` — walks `entities/*/` generically, so it lints a new `solutions/` type automatically: **orphan check** (no inbound edge / not in index), **broken-edge check** (`edge_target_exists`), and **required-frontmatter** (`id, type, title`). The opportunity↔solution bidirectional links ride the existing edge mechanism (inbound/outbound counting) — so "orphan solution (no parent)" is essentially the existing orphan/edge check.
- `nanopm_state_log` — the `shortlisted`/`chosen` status transitions log through the same typed-decision layer already used for the PRD `bet`/`prd` records.
- `nanopm_opportunity_slug` — generic slug logic; reuse directly (or lift to `nanopm_entity_slug`).

**Net-new (the real engine tasks for /pm-breakdown):**
1. `solutions/SCHEMA.md` + a `nanopm_solutions_schema` lib helper (mirror `nanopm_opportunities_schema`); defines solution-specific frontmatter (`opportunity, status, lens, appetite, impact, provenance`) on top of the generic `id/type/title`.
2. A solutions INDEX regenerator — either generalize `nanopm_opportunities_reindex` (currently hardcoded to `entities/opportunities`, grouped by theme/priority) to take a dir + grouping, or add a parallel `nanopm_solutions_reindex` grouped by parent opportunity / status.
3. Register the type: add `"solutions"` to `ENTITY_ORDER` in `bin/nanopm-ingest-agent` (index placement) and to `ENTITY_TYPES` in `bin/nanopm-migrate-to-wiki` (scaffold).
4. The two **semantic** lint rules that aren't free: solution **missing assumption/test** (body/field check) and **`chosen`-without-PRD** (cross-entity edge check) — net-new logic in `nanopm-lint-agent` (JUDGMENT-style checks).

## The One UX Decision

**How much of the panel's raw thinking does the founder see?**

- **Option A — surface the room.** Show each lens's raw proposals separately (Eng said X, Design said Y, Business said Z, with their disagreements) *then* the converged comparison table. Preserves the cross-functional tension that is the entire point — the founder sees *why* the eng-cheap option and the business-big-bet option conflict. Cost: more output, higher cognitive load.
- **Option B — surface only the verdict.** Show just the converged, deduped comparison table. Clean and fast, but it hides the debate — and a deduped table with no visible dissent is dangerously close to the single-generator "ChatGPT, give me 5 features" failure mode we explicitly rejected.

**Recommendation:** a middle path — the converged table is canonical, but each solution row keeps its originating **lens tag** and a one-line **dissent/tension note** ("Eng: cheap but caps scale at ~1k users"). The room stays visible without dumping three raw transcripts. This is the load-bearing call: get it wrong toward B and the multi-lens panel becomes indistinguishable from a single generator, invalidating the feature's reason to exist.

## Resolved decisions

*All three open questions resolved by the founder on 2026-06-26 — none block the build.*

- **Lenses are a FIXED set for v1** — Eng / Design / Business, hardcoded in the skill. ORG/personas-derived lenses (e.g. dropping Design for a repo with no designer) are deferred to a later iteration. Simpler prompts, predictable panel.
- **No status gate** — `/pm-solutions` runs on an opportunity in *any* status; it does not require `ready-for-solutions`. When solutions are written and the opportunity isn't yet `ready-for-solutions`, advance it to that status.
- **Viewer table stays in v1** — the anti-goal exception is founder-ratified (see Founder decision under Reviewer notes).

## Reviewer notes

*Advisory — surfaced by the /pm-prd review panel (solo-fast: non-blocking, the founder decides). They converge on one theme: v1 is broad. Resolve or consciously accept before handoff.*

> **Founder decision (2026-06-26): accepted v1 as scoped — viewer table and both cross-skill contracts stay in v1.** Rationale: ship the full Opportunity→Solution loop in one shot rather than dribble it. The notes below are kept as *known risks to manage during build*, not unresolved blockers. **Mitigation to honor:** build the divergence/convergence *core* first and do not let the viewer/contracts starve it (appetite-scope); and Theo's CLI path must stand alone without the viewer (persona-fit).

- **appetite-scope:** For a solo-fast appetite, v1 spreads across a new skill, 3 concurrent lens-subagents, a convergence pass, a new wiki entity with a status lifecycle, 3 lint rules, a new viewer table, AND edits to two existing skills — the cross-skill contracts and the viewer could each be their own iteration, so the comparison-engine *core* risks being under-built to fit everything in. **Suggested cut line:** v1 = skill + entity + convergence core; the viewer table and the `/pm-prd`/`/pm-opportunities` contracts as a fast-follow.
- **persona-fit:** The viewer solutions table serves the unvalidated, non-terminal Dani persona; Terminal-native Theo (the validated primary, who lives in the CLI) gets zero incremental value from that half of the build — and the plan explicitly says no viewer polish beyond a NOW/NEXT item. This is the sharpest reason to consider splitting the viewer out of v1.
- **success-measurability:** Criterion 1 (">=3 solutions per run") only verifies output *volume*, not the claimed behavior ("generates a compared set *before* speccing") — "before speccing" has no ordering check a stranger could verify. Tighten to: a PRD's `chosen` solution must reference a `solutions/` set whose files predate the PRD file (git-observable ordering).
